# 2026-05-16 — Chess experiment: tooling, architecture, and methodology

Worked through with Claude (Opus) the full implementation surface for [[experiment-chess]]: game engine, UI, backend architecture, logging schema, tool-fairness rules, and the white-only methodology question. No code written yet — this entry records the settled design so the build can start cleanly from here.

## Game engine and standards

The chess world has well-established standards; no custom rules engine is needed.

- **FEN** for single positions, **PGN** for full games, **SAN** for human-readable moves, **UCI** for engine I/O. Anything we build is in terms of these.
- **python-chess** is the obvious library. Covers init, move-making, legal-move generation, all status detection (check / checkmate / stalemate / insufficient material / threefold / 50-move), PGN read/write, and UCI engine driving via `chess.engine`. Settled.

For game analysis export, PGN is the lingua franca. Lichess has `/api/import` and `/api/cloud-eval`; local Stockfish via python-chess gives programmatic analysis without rate limits. Probably both are useful — Lichess for sanity-check and shareable URLs during development, local Stockfish for the actual learning loop.

## UI layer

We surveyed the options:

- python-chess's `chess.svg` is static-only, not interactive.
- **chessboard.js** (MIT) — simple, classic.
- **chessground** (GPL) — the board widget Lichess itself uses. Headless on rules, expects the backend to provide legal moves. Polished UX (legal-move dots, arrows, premoves).
- **react-chessboard** (MIT) — React component.
- Full GUIs (PyChess, CuteChess, SCID) were considered and rejected. PyChess is heavy and hard to modify; CuteChess is engine-vs-engine only via UCI and doesn't accommodate human-in-the-loop or LLM agents without wrapping the agent as a UCI engine.

Settled on **chessground**. The pickier setup is worth the visual quality, and the headless-on-rules design fits the architecture below cleanly.

## Backend architecture — the key decision

My first instinct was for the frontend to be the orchestrator: it would pick between agent-API and bot-API per move, push the chosen move to the python-chess server, and pass moves back. Claude pushed back hard on this and was right. The reason: **Phase 1 is fundamentally headless.** A matchmaking pool of hundreds or thousands of games can't depend on a running browser tab. So orchestration belongs in the backend.

Settled architecture:

- **Backend (FastAPI + python-chess)** owns the game state and the `white_player` / `black_player` config. It drives the move loop, calling out to whichever Player implementation each side is configured to use.
- **Player abstraction** with three implementations:
  - `HumanPlayer` — blocks on a queue that the frontend fills via POST.
  - `EnginePlayer` — wraps a UCI engine via python-chess's `chess.engine`.
  - `AgentPlayer` — hits the agent's API (my existing harness, or invoked through Claude Code).
- **Frontend (chessground)** is an optional adapter. It subscribes to the game's SSE stream for board updates, and if a local side is configured as Human, it emits move attempts via POST. For training runs (headless Phase 1) there is no frontend at all — same backend.
- **Agent reasoning side-channel** — separate SSE stream from the agent harness for visualising thought/tool-use in the UI. This is for inspection only, not for moves. Moves go directly from the agent to the game API.

Game creation is just config: `POST /games { white: "agent", black: "stockfish-1500" }`. Switching between the three player types is per-side.

The harness's client-side-function feature (where it can emit "request the client to do X" via SSE rather than running a function itself) is useful for the agent's *side channel* — e.g., "show me this hypothetical position on a side board" — but not for the agent's actual move submission. Moves should be direct so they don't require a frontend.

## Logging: files-in-git, queried with pandas

Considered SQLite vs CSV vs structured files in git. Settled on the third option because the data is part of the thesis artefact and should be git-diffable and reviewable without tooling.

Schema:

- `games/games.csv` — one row per game. Columns: `game_id`, `batch_id`, `datetime`, `opponent` (model + Elo), `result` (win/loss/draw), `elo_before`, `elo_after`, `skill_repo_sha`, `model`, `temperature`.
- `games/<game_id>.pgn` — the full PGN, one file per game. Git-diffable, viewable in any chess tool.
- `games/<game_id>.json` — agent reasoning trace (tool calls, skills loaded, decisions made).
- `diary/<batch_id>.json` — structured diary entries per batch. Reference games by ID, record lessons learned, actions taken, mistakes encountered, hypotheses tested.

Querying is via **pandas**. (Claude initially suggested duckdb; I prefer pandas since I'm already using it elsewhere.)

### What `skill_repo_sha` is and why it matters

The agent's skill library is its own git repo. `skill_repo_sha` is the commit hash of that repo at the moment the game was played. This is what makes the experiment reproducible and reversible: `git checkout <sha>` recreates the exact agent that played game N. It's also what makes "which actions caused the biggest Elo jump" a real question — diff SHAs between consecutive batches, look at the batch diary, see what changed.

### Why we're not logging random_seed

Claude raised it; I pushed back. With fixed model and fixed temperature, seed is theatre because most inference APIs (Anthropic's included) don't expose or honour a real seed. You can't reproduce the inference anyway. Logging model + temperature is enough; seed is dropped.

### What a batch is

A batch is the set of games played at one `skill_repo_sha`. Between batches is where the self-correction step happens — the agent updates its skill library, the SHA changes, a new batch starts. This makes "batch" the natural unit for analysing the learning loop, not just an arbitrary session boundary.

### Current Elo

Computed from `games.csv` — last row's `elo_after`. No separate storage. Duplicating invites drift; at thesis scale (probably <10k games) recomputation is free.

## Tool fairness — the rule we settled on

The methodological question was: which tools can the agent use without making the experiment cheat? Initial framing was "what could a human reasonably do," which is too vague. The sharper rule we settled on:

> **Mechanics tools are always fine. Retrieval tools are fine if-and-only-if the corpus they retrieve over was built through the learning loop.**

Concretely:

- **Mechanics tools (always fine):** `legal_moves()`, `what_if(move) -> board_after`, `is_attacked(square)`, `piece_values`, `list_pieces_attacking(square)`, board geometry checks for back-rank patterns or pin detection. These read the board with no chess understanding baked in.
- **Retrieval tools (fine iff the corpus is agent-curated):** `search_my_notes(position)`, `retrieve_opening_theory(moves_so_far)`. Smart retrieval (vector search, indexing, semantic similarity) is permitted — the question is *what* the retrieval runs over, not how clever the algorithm is. A human GM with an organised notebook doesn't sequentially scan every page; they use an index. The deterministic-tool equivalent is the same kind of efficient access to one's own knowledge.
- **Retrieval tools (cheating):** pre-loaded vector DBs of "every position from these 10 books" without the agent having processed them through the loop. Pre-curated Stockfish evaluation databases. `recognize_opening()` backed by an externally-built encyclopedia. Even if the retrieval algorithm is trivial, the corpus is contamination.

Subtle case: the agent reads a chess book during Phase 1 and writes its own notes about positions into its wiki. Those notes become the agent-curated corpus even though the source material was external. The act of reading-and-noting is part of the loop. What's not allowed is dumping book content into a vector DB the agent has never processed.

A second subtle point — purely geometric pattern detection (e.g., "is this position a back-rank mate pattern?") with no corpus reference is a *mechanics* tool, same category as `list_pieces_attacking`. But "is this position from the Najdorf?" is encyclopedic and only allowed if "Najdorf" exists in the agent's own wiki.

### Stockfish and Lichess: post-game only

The agent can use Stockfish and Lichess for *game analysis* — post-game reflection, synthesising what went wrong, building skills/notes from the analysis. It cannot call Stockfish *during* a game. This keeps the deterministic-tools-hypothesis test (Phase 2 Config 3 = full engine during play) uncontaminated by accidental engine assistance in Configs 1 and 2.

### Calculation tools

Allowed, with one caveat: **the agent has to drive the search itself**. Tools like `what_if_i_play(move) -> board_after` are fine; the agent decides whether to recurse manually. A tool that returns "the best 3-move sequence from here" is a chess engine with the interface filed off and is not allowed.

## Parallel LLM wiki for the agent's chess knowledge

The agent's accumulated chess knowledge should be structured as its own LLM wiki — same architectural pattern as this thesis knowledge base, different schema. Probably something like:

- `openings/` — synthesised opening theory pages the agent has built.
- `patterns/` — tactical and positional patterns.
- `endgames/` — endgame principles.
- `game-analyses/` — per-game post-mortems.

The principle of "compiled synthesis pages owned by the agent, with backlinks and sources from raw materials (books, prior games, post-game analyses)" transfers cleanly. This gives the thesis a bonus result: a second-domain test of the [[karpathy2025wiki]] / LLM Wiki pattern alongside this one. Worth being explicit about that in the methodology section.

It also incentivises the agent to organise what it learns into reusable concept pages rather than dumping everything in one giant skill file.

## White-only — settled across both phases

Initial proposal: limit Phase 1 to white-only to halve the learning surface. Defensible because the thesis claim is about the self-correcting loop, not about chess mastery.

Concern I raised: in Phase 2 we compare three configurations, and if only Config 2's skill library is "trained white-only," giving it the white side in head-to-head matches looks like an asymmetric advantage. We worked through the options:

- **Option A:** White-only across both phases. All three configs play white against a fixed pool of black-playing bots (Maia or Stockfish-at-Elo). Elo per config against the shared pool. Configs ranked by Elo.
- **Option B:** White-only Phase 1, both colors Phase 2. Quietly broken — Config 2's skill library has nothing for black, so on black games it behaves like Config 1, and the "skill library contribution" gap collapses on half the games.
- **Option C:** Both colors in Phase 1. Doubles the learning surface and arguably doubles Phase 1 duration. Risk-register flagged this.

**Settled on Option A.** The white-advantage cancels across configs because all three get it equally. The skill-library contribution (1 vs 2) and the deterministic-tool contribution (2 vs 3) are measured under matched conditions. What we give up is direct Config-vs-Config head-to-head matches; what we keep is the ranking via shared-opponent Elo.

Methodological consequence: Phase 1's matchmaking pool and Phase 2's tournament pool can be the *same* pool. Phase 1 builds the library against pool X; Phase 2 freezes the library and measures all three configs against pool X. Clean.

Caveat to note in write-up: white-only Elo against a fixed bot pool is *internal* Elo and not directly comparable to FIDE or Lichess Elo. Internal comparison across configs is what the thesis claim needs, and that's preserved.

## Other decisions

- **Time controls excluded.** The agent runs inference through APIs, so its move time is dominated by network and server compute, not by chess-skill management. Including time would add complexity without measuring what we care about.
- **Opponent pool:** **Maia chess bots** are preferred over Stockfish-at-Elo for Phase 1 matchmaking because Maia is trained to make human-like mistakes at calibrated Elo bands (1100, 1300, 1500, 1700, 1900). This is methodologically more defensible than fighting a weak Stockfish, which makes alien-looking mistakes. Stockfish at varying skill levels still in play as a complementary or fallback option.
- **Phase 2 Config 3:** full-strength Stockfish (not Elo-limited) as the deterministic upper bound.

## Architectural pattern worth flagging

If we wrap the LLM agent as a UCI engine (UCI is just a stdin/stdout text protocol), it can plug directly into **cutechess-cli** or **fast-chess**, which handle tournament management, opening books, time controls, SPRT testing — all out of the box. Currently we plan to build our own orchestration in the FastAPI backend, which is right for Phase 1 (where we need fine-grained control over the learning loop and the agent's side-channel). For Phase 2 it may be worth re-evaluating whether to use cutechess-cli for tournament management. Decision deferred until Phase 1 is running.
