# 2026-06-02 — Typed-tool harness becomes the standard execution model

## What happened

The typed-tool harness reached a verified, working state today. The chess experiment ran end-to-end against the new implementation on eX3 (Gemma 4 31B, hgx2q, TP=2) without issues. `run_script` is gone. Every tool call the model makes is now in the native tool-calling format.

## Why this matters beyond chess

The harness was motivated by the chess experiment, but what was built is a general mechanism. Any skill with scripts now gets typed, individually-named tools registered at agent build time, hidden behind a `prepare` gate until the skill is loaded. The model never sees an untyped dispatcher; it sees exactly the tools that are available to it at that moment, in the schema it was fine-tuned on.

This is the execution model I expect to use for every subsequent experiment. The pattern is stable enough to treat as settled infrastructure:

- Schema extraction at parse time (argparse → JSON Schema via static AST analysis)
- Progressive disclosure: `prepare` gate on `activated_skills`, re-evaluated per step
- `activated_skills` persists across turns, resets only on `clear_conversation()`
- Subprocess execution with correct `cwd` and `PYTHONPATH` — sibling imports work
- The typed tools coexist with `read_reference` and other skill tools; the model chooses

## Architectural decision

This is a durable architectural decision, recorded at `decisions/2026-06-01-typed-tool-harness-spec.md`. The key resolution from today's implementation and verification:

The `prepare` gate does not require a `ModelRetry` to activate. pydantic-ai re-evaluates `prepare` on every step (each `CallToolsNode → ModelRequestNode` transition). When `use_skill` returns normally, the tool-call loop produces the next model request automatically — by that point `activated_skills` already includes the new skill, and `prepare` opens the gate. No fake error in history, no retry-budget consumption.

## What I am watching going forward

Two things introduced today that any future experiment should know about:

1. **Prefix-cache bust**: Activating a skill changes the tool set sent to the model. On vLLM, this busts the KV prefix cache from that point in the prompt — a one-time latency hit per `use_skill` call, not per turn. Acceptable cost, but worth knowing.

2. **`activated_skills` persistence**: Skills loaded in turn N are visible in turn N+1. This is intentional (skills should stay available once loaded) but it is a semantic change from the previous behaviour where the list reset on every `run()` call. Future experiments should not assume skill state resets between prompts unless `clear_conversation()` is called.
