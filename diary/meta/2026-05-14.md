# 2026-05-14 — Designing the knowledge base

Spent a long session with Claude designing the knowledge base that will track this thesis project. Not the experiments themselves — those live elsewhere — but everything around them: literature, decisions, reflections, evolving synthesis. The pattern is Karpathy's LLM wiki ([[karpathy2025wiki]]), but adapted for an academic project with substantial user-authored content. This entry captures the discussion and the design we settled on, because the reasoning is going to fade and I want it preserved.

## Starting point

I came in with rough ideas: a `literature/` folder for sources with citations and summaries, a `projects/` folder for experiments and modules, a `diary/` for daily reflections, and some kind of `meta/` folder for deadlines and plans. The knowledge base would live alongside other projects.

Open questions I wanted to bang out:
- Do PDFs need markdown equivalents, or can the maintaining LLM read them directly?
- Diary cadence — per session, per project, per day?
- Do I need a `raw/` folder if the whole thesis repo could serve that role?
- How do I handle source files that change?

## Decisions, in roughly the order they got made

**PDFs stay as PDFs.** Modern agents read them fine. Translating them to markdown produces a derivative that drifts from the source, and the actual "translation" you want — a per-source summary with relevance to the project — is a different artifact entirely. Lives in `wiki/literature/<citekey>.md`, not as an md mirror in `raw/`. The one exception is scan-only PDFs with no OCR layer; those need a one-time text extraction stored alongside the PDF.

**BibTeX, centrally.** One `references.bib` at `raw/literature/`. Each wiki literature page carries the citekey in frontmatter. Avoids per-paper `.tex` snippets.

**Diary: per working day.** One file per day I actually worked on the thesis. Per-session is too granular (I'll abandon it). Per-project is too coarse (loses the temporal thread). Multi-project days link out to the relevant work pages.

**Keep `raw/`.** Two reasons: the immutability invariant is load-bearing — the LLM needs a bright line between what it can edit and what it never touches. And experiment code churns constantly; if the wiki treated the whole repo as raw source, every commit would trigger re-ingestion. Better to keep the knowledge base in its own folder with a clean `raw/` zone, and have experiment documentation in `work/` link out to wherever the code actually lives.

**Editing sources.** Published papers in `raw/literature/`: never edit. Your own writing in `raw/interviews/` (transcripts) once finalized: same. The "immutable" property is about not corrupting the evidentiary basis — typo fixes and version replacements are fine, just note them in `log.md`. Git handles version history; date-stamping inside `raw/` is overkill.

## The mental model that took the longest to settle

Karpathy's framing collapses everything into "raw" vs "wiki" — sources vs LLM-generated synthesis. That breaks down for an academic project because there's a huge third category: my own authored content. Diary, decisions, experiment notes, scope choices, ideas. None of that is external evidence (so not `raw/`), and none of it should be LLM-owned (so not `wiki/`).

The right model is three roles, not two:

1. **External source of truth** — papers, finalized interview transcripts, datasets. I didn't author them; they don't change. Lives in `raw/`.
2. **User-authored content** — diary, admin, work, decisions, archive, manuscript-notes, inbox. I authored these; I edit them freely. The LLM reads them as input but doesn't own them.
3. **LLM-compiled synthesis** — literature summaries, concept pages, the index. The LLM owns these. I read them.

The wiki is *one of three zones*, not a container for everything. This is the cleanest version of the model. The top-level folder structure encodes ownership, not content type.

Wiki pages can — and often will — contain information that has no source in `raw/`. A concept page on the deterministic-tools hypothesis draws mostly from my own thinking, diary entries, and experiment results. `raw/` is just the subset of inputs that happens to be external and immutable. The wiki isn't an index of raw sources; it's compiled synthesis, and the inputs to synthesis are everything.

## Mutability — how it actually works

I was confused for a while about how the wiki handles change. The model that snapped into place:

- **Raw sources are immutable by convention** (not by law). Don't rewrite them to change what they say. Typo fixes, version replacements, OCR extractions — all fine, just log them.
- **User-authored files are mutable.** I edit them directly. When something changes, I tell the LLM, or the LLM notices on its next pass and re-reads.
- **Wiki pages are mutable, owned by the LLM.** It rewrites and restructures them as the synthesis evolves.

A single user action — "I made progress on the math experiment" — typically touches 10-15 files. The LLM updates `work/experiment-math.md`, propagates to `wiki/concepts/calibration-thread.md` and `wiki/concepts/deterministic-tools-hypothesis.md`, possibly updates `wiki/literature/kalai2024hallucinate.md` with a backlink, appends a note to today's diary, logs the operation. That multi-file propagation is the whole point.

I asked: can I just say "remember this thought" and have it filed? Yes. The LLM decides where: one-liners to `admin/ideas.md`, substantial ideas to a draft concept page, decisions to `decisions/`, reflections to `diary/`. If unsure, it asks rather than guessing.

## Where things go — the coverage check

Worked through this explicitly to make sure nothing's missing a home:

- Paper I'm citing → `raw/literature/` (PDF) + `wiki/literature/<citekey>.md` (LLM summary)
- Interview transcript (finalized) → `raw/interviews/` + `work/interview-<name>.md` (working notes, my authorship)
- Experiment design and progress → `work/experiment-<name>.md`
- Shared module used by multiple experiments → `work/module-<name>.md`, linked from each experiment
- Useful external repo I'm referencing → `work/external-<name>.md`
- Daily reflection → `diary/YYYY-MM-DD.md`
- Cross-cutting idea (calibration, deterministic-tools, etc.) → `wiki/concepts/`
- Connection I just realized → `wiki/concepts/` (new page or update)
- Scope/architecture decision → `decisions/YYYY-MM-DD-slug.md` (ADR pattern)
- Finding from an experiment → `work/experiment-<name>.md` + propagate to relevant `wiki/concepts/`
- Deadline or supervisor meeting note → `admin/`
- Half-baked idea → `admin/ideas.md`
- Something I dropped → `archive/`
- Notes for writing a thesis chapter → `manuscript-notes/`
- Quick capture I haven't sorted yet → `inbox/` (added later in the conversation)

`work/` is flat — experiments, modules, external repos, and interview working-notes all live as siblings, disambiguated by a `type:` frontmatter field. Moving things between types as the project evolves should be easy.

I considered `scratch/` for half-formed exploration but dropped it; `admin/ideas.md` covers one-liners, and substantial drafts can sit in `wiki/concepts/` with `status: draft` in frontmatter.

## Why `wiki/literature/` exists but there's no `wiki/work/`

This came up late in the discussion and is worth recording because the asymmetry is real and intentional.

External sources need both a raw form (the PDF) and an LLM summary (the wiki page) because the source itself isn't tailored to my project. The summary page is where "this paper proves hallucination is structural, which is why our calibration thread isn't just engineering" gets written down. The PDF doesn't say that; the wiki page does.

My own experiments don't have that separation. I am the source. `work/experiment-math.md` already contains my thinking, written for my project. An LLM-owned "synthesis page about my experiment" would just be a rewrite of my own document — pointless and risky (the LLM might "improve" my in-flight reasoning).

What the wiki does for experiments instead: pulls findings into concept pages. `wiki/concepts/calibration-thread.md` cites `[[experiment-math]]` and synthesizes what the experiment found alongside what Kalai proved. The cross-cutting synthesis happens in `wiki/concepts/`, not in a parallel `wiki/work/`.

## What Claude found from looking at how others use Karpathy's pattern

Most of the secondary writing online is rehash. Three things worth keeping:

- **Link integrity is a real problem.** When the LLM renames or restructures pages, backlinks rot silently. Mitigation: lint workflow verifies all wikilinks resolve, not just orphan-page detection.
- **Lex Fridman's "ephemeral wiki" pattern** — spawning a focused sub-wiki for a deep session. Could be useful when I'm deep in an experiment; not building it upfront.
- **DAIR.AI has a Wiki Builder plugin for Claude Code.** Skipping it — want full control over the schema for an academic project.

## The pivot from "tracking the project" to "the project is the case study"

I framed this as "build a knowledge base to keep track of the thesis project itself, not for the experiments." Claude pushed back on the underclaim. The thesis is literally about the infrastructure that makes AI agents productive on real-world tasks. Building a working knowledge base for the thesis work *is* a case study of the framework — particularly the Knowledge component (persistent storage and retrieval across sessions) and arguably the Skills component (the `CLAUDE.md` schema is a skill in the Zhang et al. sense, encoding workflows progressively disclosed to the agent).

Not committing to writing it up. But I'm going to tag relevant diary entries `meta-observation` from now on, so if the wiki succeeds or fails in interesting ways the evidence is preserved.

## The final structure

```
knowledge-base/
├── CLAUDE.md                # schema and workflows
├── AGENTS.md                # symlink to CLAUDE.md
├── index.md                 # LLM-maintained catalog
├── log.md                   # LLM-maintained chronological log
├── raw/
│   ├── literature/          # PDFs + references.bib
│   ├── interviews/          # finalized transcripts
│   └── datasets/
├── wiki/
│   ├── literature/          # one summary page per source
│   └── concepts/            # synthesis pages — the compounding value
├── work/                    # experiments, modules, external repos, interview working-notes
├── diary/                   # YYYY-MM-DD.md
├── admin/                   # deadlines, supervisor notes, ideas.md, todos
├── decisions/               # ADR-style records
├── archive/                 # abandoned experiments, scrapped drafts
├── manuscript-notes/        # LLM-readable thesis-writing context
└── inbox/                   # capture surface (added in patch)
```

`AGENTS.md` is a symlink to `CLAUDE.md` so agents other than Claude Code (Codex, OpenCode) pick up the same schema without duplication.

## Operations the LLM performs

Settled on four, then added a fifth:

1. **Ingest** — user adds a source to `raw/`, LLM reads it, writes the wiki summary, updates relevant concept pages, updates index, logs.
2. **Capture** — user says "remember this" or "log progress"; LLM files the thought to the right destination per the rules above and propagates.
3. **Query** — user asks a question; LLM reads index then drilled pages, synthesizes with citations, offers to file the answer back as a new concept page.
4. **Lint** — periodic health check: link integrity, orphans, stale frontmatter, contradictions between pages, missing concept pages, status drift.
5. **Process inbox** — read files dropped in `inbox/`, decide destination per Capture rules, propagate, log, **delete the inbox file** on successful processing. If ambiguous, leave it and ask.

The deletion in step 5 is the key design property — keeps `inbox/` doing exactly one job (capture surface) and forces chain-of-custody through `log.md` rather than a parallel "processed" folder. I had originally proposed `mail/inbox` and `mail/processed`; Claude argued against the processed folder on duplication grounds. Convinced.

## Frontmatter

Every wiki page and every `work/` file has YAML frontmatter. Required fields differ by type, but the common spine is `type`, `status`, `updated`, and the appropriate `related_*` arrays for backlinks. Linking is via wikilink syntax `[[citekey]]` or `[[slug]]`, resolved during operations. Backlinks are maintained implicitly through `related_*` fields rather than separate backlink sections.

## What I want to remember about this design

The thing I almost missed: the top-level folder structure encodes *ownership*, not content type. Putting `diary/` inside `wiki/` would tell the LLM "this is yours to restructure," which is wrong — diary entries are evidence of what I thought on a given day, and their value is partly that the LLM didn't write them. Same for decisions, admin notes, experiment documentation. The wiki is one of three zones, not a container.

The other thing: the wiki contains real content. It's not just nodes and edges. Concept pages have prose, synthesis, analysis. The wiki *links to* content in other folders, but it has substantial content of its own — that's where the compounding value lives.
