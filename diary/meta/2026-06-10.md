# 2026-06-10 — Testing Fable 5 on a large, loosely-specified task

*Entry written by Claude (Fable 5) itself, at Andreas's request: he is using
this session to see how the new model handles a big multi-part task with
minimal supervision. That makes this entry an unusual artefact — the test
subject reporting on its own test — so read its self-assessments with that
in mind. The verifiable record is the git history and the diary entry at
[[diary/experiment-chess/2026-06-10]].*

## The setup

One prompt, ~4000 characters, delegating an entire branch of the chess
experiment: ingest theory sources into the agent wiki (including converting
a PDF of diagrams into FENs), redesign per-turn memory, add perception
tools without breaking the fairness rules, build a way to test on specific
positions, run live sessions on eX3, keep commits atomic and revertible,
write the diary, open a PR. Constraints stated once: fairness rulebook is
binding, prefer simple mates over fancy ones, the reader is a much weaker
model than the writer, no calibration runs on main.

## What the delegation produced (verifiable)

Seven commits on `mating-patterns-and-strategy` over one session: a
FEN-verified Capablanca conversion, nine wiki pages, a mate/draw radar with
20 tests, the structured turn memory with 18 tests, puzzle mode with API
tests, two pre-existing test failures repaired, and live puzzle sessions on
eX3. One ADR in the knowledge base. Every chess claim that entered the
corpus was machine-checked in python-chess rather than trusted from
training data — including catching its own wrong-side-luft example before
commit, and a probable century-old notation subtlety in Capablanca's
Example 16.

## Observations worth keeping (from inside the run)

- **The most valuable find was nobody's request.** The system-prompt-loss
  bug (turns 2+ played with no system prompt since 2026-05-26) surfaced
  only because the memory redesign required understanding pydantic-ai's
  injection semantics deeply enough to probe them empirically. Delegated
  refactors that *touch* old code seem to audit it for free.
- **Verification compounds trust.** The instruction "the agent trusts these
  pages" turned every wiki example into a python-chess check; three errors
  were caught that way. A reviewer can now audit the corpus by re-running
  scripts instead of re-deriving chess.
- **The fairness rulebook worked as a design tool, not just a filter.** The
  radar was shaped *by* the mechanics-tools clause (geometry yes, scoring
  no; pointers into the agent's own corpus yes, oracle moves no). Having a
  citable ADR made those calls fast and defensible.
- **Friction points were environmental, not conceptual:** a dropped SSH
  tunnel (required Andreas), `gh` not installed (PR path needs a human or
  an install), and one path quirk (the em-dash venv workaround documented
  in the repo's CLAUDE.md was load-bearing).
- **Where guidance was thin, the work followed the failure data.** The page
  selection, the radar's content, and the plan-channel design all trace to
  three drawn games rather than to the prompt — the delegation worked
  because the repo carries its own evidence (logged games, ADRs, diary).

## For the thesis

This session is itself a data point for [[agent-infrastructure-vs-capability]]:
the same delegation a year ago would have needed the human in the loop for
the PDF diagrams, the notation archaeology, and the bug hunt. What made it
tractable was less raw capability than the *infrastructure around the task*
— the knowledge base's operating manual, the experiment's ADR trail, the
auditable game logs, and a venv with python-chess in it. The model could
check itself because the project was built to be checkable.
