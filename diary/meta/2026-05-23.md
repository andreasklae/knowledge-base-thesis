# 2026-05-23 — eX3 inference server: architecture and access pattern settled

## Summary

Spent the session designing the self-hosted LLM inference setup on [[ex3]] — both for the experiments that need a controlled inference endpoint and for general thesis tooling (Cursor, Pydantic AI agents). Started from a half-formed plan (Ollama + FastAPI + Docker, persistent server, API-key auth on the open internet, dynamic GPU allocation) and ended with a substantially different one (vLLM + on-demand Slurm jobs + SSH tunnel + local CLI, no exposed endpoint, no daemon on eX3). Decision recorded in [[2026-05-23-ex3-llm-inference-server-architecture]].

## What I came in with

The original framing was:

- Host a backend server on eX3 to serve LLM inference.
- Wrap an Ollama model behind FastAPI.
- Run inside Docker, pull image from GitHub.
- Keep it always running 24/7, with "dynamic" GPU allocation only when inference is actually requested.
- Expose on the internet with API-key auth.
- Use it from Cursor (custom OpenAI base URL) and from Pydantic AI agents.

The administrator's verbal approval was a vague yes when I asked about hosting a server. I wasn't sure whether that extended to "always-on personal API endpoint."

## What the discussion changed

### 1. "Dynamic GPU allocation" is not how Slurm works

Pushed back hard on this and rightly so. eX3 is Slurm-managed: a GPU allocation is for a fixed window (`--time=NN:NN:NN`). The GPU is yours for that window whether you're inferencing or idle. Ollama unloading the model from VRAM after inactivity doesn't release the allocation back to Slurm. So "only consume when needed" only works if "needed" is the unit of a job, not the unit of an HTTP request.

This collapsed the always-on plan. A 24/7 endpoint either holds a GPU continuously (violates "don't let allocations sit idle") or has no GPU when a request arrives (defeats the purpose).

### 2. The compute nodes have no public IPs

I'd been assuming the "expose on the internet with API keys" model. Compute nodes on eX3 only reach the outside world via the login node — they aren't directly addressable. So the right access model isn't "public service with auth" but "SSH tunnel from my laptop, service bound to localhost." That removes the entire auth-exposure surface and is also a better security model (nothing else can reach the service).

This also rules out, e.g., letting a teammate hit the endpoint. Fine for solo thesis work; would be a different problem otherwise.

### 3. Ollama is the wrong inference engine for this hardware

The eX3 docs already recommend vLLM. vLLM is built for data-center GPUs (batched scheduling, paged attention, tensor parallelism), exposes a native OpenAI-compatible API at `/v1`, and would actually use the A100 — Ollama would leave significant throughput on the table. There's no real reason to pick Ollama here except familiarity. Switched to vLLM.

The OpenAI compatibility is also what makes Cursor + Pydantic AI trivial: both already speak the OpenAI REST shape, so pointing them at `http://localhost:11434/v1` with a placeholder API key works without any glue code.

### 4. The "lightweight always-on controller daemon" idea

I floated: maybe a tiny 24/7 service on the login node with a `/start` endpoint that allocates a GPU for N hours on demand and dies. Reasoning: only the controller is always on, the actual GPU is held only for the requested window.

Rejected for three reasons:
- Login nodes are shared infrastructure, not meant to host user-owned long-running services. Admins notice.
- Login nodes get rebooted; the daemon dies; back to manual.
- It adds an auth/exposure surface (the `/start` endpoint itself) for zero benefit over just running `ssh ex3 sbatch llm.sbatch` from the laptop.

The right shape is a local CLI on my laptop. No always-on process on eX3 at all.

### 5. Local dev assumption was wrong

I'd implied "write the code locally" — which is fine for editing/pushing — but in fact vLLM can't run on a typical laptop (no CUDA on M-series Mac; consumer NVIDIA GPUs can't host the model sizes we care about). So there's no local execution loop. Bootstrap has to happen on eX3 directly. The clarification was that I meant authoring, not testing.

## What we landed on

The architecture, in one breath: server code lives in a GitHub repo cloned to `/global/D1/homes/$USER/llm-server/` on eX3. Models live in `/global/D1/homes/$USER/models/` on BeeGFS. A local CLI on my laptop SSHes in, submits a time-bounded `sbatch` job that starts vLLM on a high port (e.g. `28811`) on the assigned compute node, polls `squeue` until `RUNNING`, then opens a two-hop SSH tunnel from `localhost:11434` on the laptop through the login node to the compute node's vLLM port. Cursor and Pydantic AI point at `http://localhost:11434/v1`. When done, the CLI cancels the job and closes the tunnel. The Slurm time limit acts as a hard fail-safe if I forget.

The full architecture spec is in [[2026-05-23-ex3-llm-inference-server-architecture]].

### Why `/v1`

Side question that came up: the `v1` in the URL is OpenAI's API version prefix (`https://api.openai.com/v1/chat/completions`). vLLM mirrors that path structure exactly so any OpenAI client just works by swapping the base URL. Not magic — just convention.

### Port choice

`8000` is too common (lots of dev tools bind there) — switched to a high uncommon port (`28811` remote, `11434` local). `11434` is Ollama's default so it's familiar muscle memory for "local LLM port" without actually running Ollama.

## Plan from here

Four phases, in order:

1. **Bootstrap on eX3 directly with a tiny model.** Get an interactive `srun` GPU session, install vLLM in a venv, download `Qwen/Qwen2.5-0.5B-Instruct`, run vLLM by hand, `curl /v1/models` inside the session, confirm it answers. Don't write any CLI yet — confirm the mechanism on real hardware first.
2. **Repo + sbatch.** On the laptop, create the repo: `start.sbatch`, `requirements.txt`, `config.yaml` (model name, port, api key var), README. Push to GitHub. `git clone` it onto eX3 and reproduce phase 1 via `sbatch start.sbatch` instead of manual commands. Confirm logs go to `<jobid>-<jobname>-stdout.txt`.
3. **Local CLI.** Write `llm-up`/`llm-down`. Logic: ssh → sbatch with `--time` → poll squeue for `RUNNING` and node name → open `ssh -N -L 11434:$NODE:28811 ex3` → poll `localhost:11434/v1/models` until 200 OK. Teardown: kill tunnel, `scancel`. Test the full flow with the tiny model and time the cold start.
4. **Real model.** Download the production model to BeeGFS (one-time multi-gigabyte from inside an `srun` session, not on the login node). Update config. Re-run `llm-up`, confirm Cursor + Pydantic AI both connect through the tunnel.

## What I'm not building

- An always-on service on eX3 (any form).
- An internet-exposed endpoint with API-key auth.
- A FastAPI wrapper around vLLM (vLLM already exposes the OpenAI-compatible HTTP server natively; a wrapper would only be needed for custom routing or extra middleware, which I don't have).
- A Docker-based deployment, unless eX3 confirms it's permitted and offers a benefit over a pinned venv.

## Afternoon: Phase 1 bootstrap on real hardware

Did phase 1 of the plan above: end-to-end vLLM + tiny model on a real GPU node, confirmed inference works. Hit several real bugs that are worth writing down so the next session doesn't re-discover them.

### What ended up working

The full happy path that succeeded:

1. SSH to login node: `ssh ex3` (already configured in `~/.ssh/config` to hit `dnat.simula.no:60441` as user `andreasklaeboe`).
2. Load Slurm: `module load slurm/21.08.8`. **Note: the wiki/ex3 page says `slurm/20.02.7` — that version is gone.** Have to `module avail slurm` to find current; today it was `slurm/21.08.8` from `/cm/local/modulefiles`.
3. Interactive A100 allocation: `srun -p a100q --gres=gpu:1 --time=01:00:00 --pty /bin/bash`. Landed on `n014` (n013/n014 are the `a100q` nodes with 2× A100 40GB each). Got JOBID 1238143.
4. Verified hardware: `nvidia-smi` showed A100 40GB with CUDA 13.2 driver. `which python3.11` → `/usr/bin/python3.11` (Python 3.11.15 system-installed).
5. Venv on BeeGFS: `python3.11 -m venv /global/D1/homes/$USER/llm-server/venv`, then `source` + `pip install vllm`. vLLM 0.21.0 installed cleanly; pip bundled CUDA 12.x libs work fine against the 13.2 driver (forward-compatible).
6. Downloaded `Qwen/Qwen2.5-0.5B-Instruct` via `git clone` from `https://huggingface.co/Qwen/Qwen2.5-0.5B-Instruct` into `/global/D1/homes/$USER/models/`. The .safetensors weights came down as a 134-byte LFS pointer (git-lfs not installed on login node), so curl'd them directly: `curl -L -o model.safetensors https://huggingface.co/Qwen/Qwen2.5-0.5B-Instruct/resolve/main/model.safetensors` → 943 MB.
7. Launched vLLM on `n014`: `python -m vllm.entrypoints.openai.api_server --model /global/D1/homes/$USER/models/Qwen2.5-0.5B-Instruct --host 0.0.0.0 --port 28811 --max-model-len 4096`. Startup took ~2.5 minutes (21 s torch.compile, ~90 s CUDA graph capture, then Uvicorn). Saw `Application startup complete.`
8. From a second SSH session, went `ssh ex3` → `ssh n014` (direct SSH from login node to compute node works, see gotchas), then `curl http://localhost:28811/v1/models` → got the JSON model list back.
9. Real inference test: `curl http://localhost:28811/v1/chat/completions` with a chat payload → returned a coherent one-sentence answer about transformers. Phase 1 complete.

So: vLLM + Qwen2.5-0.5B + A100 + OpenAI-compatible API + intra-cluster curl = working.

### What failed and how I worked around it

#### `hf` CLI silently corrupts large file downloads on BeeGFS

This burned the most time and the symptom was deeply confusing.

`hf download Qwen/Qwen2.5-0.5B-Instruct` (using HuggingFace's new `hf` CLI v1.16.1; the old `huggingface-cli` is deprecated) produced a `tokenizer.json` of exactly 8,854,365 bytes that crashed vLLM with `UnicodeDecodeError: 'utf-8' codec can't decode byte 0xc2 in position 5242879`. Position 5,242,879 is one byte short of 5 MiB — a suspiciously round offset.

Hypotheses I worked through and rejected:

1. **Truncation at 5 MiB.** Wrong — file was 8.4 MB, not 5 MB. Larger than correct, not smaller.
2. **Corrupt blob in shared HF cache (`/global/D1/huggingface/cache/hub`, populated by user `mikkel`).** Wrong — downloading fresh into a private `--local-dir` produced the same exact 8,854,365-byte file with the same byte-position corruption. Bit-for-bit reproducible.
3. **`hf_transfer` Rust accelerator chunking bug.** Worth checking; ran `pip list | grep transfer` → empty. Package wasn't installed at all. Wrong.
4. **Env var `HF_HUB_ENABLE_HF_TRANSFER=0`.** Tried setting it explicitly; no effect (because hf_transfer wasn't the cause).

What confirmed `hf` itself was the problem: `curl -L https://huggingface.co/.../tokenizer.json` produced a **7,031,645-byte** file that decoded as valid UTF-8. So the correct file is 7 MB; `hf` was producing a corrupted 8.8 MB file with deterministic corruption at the 5 MiB boundary regardless of cache config or accelerators.

Best guess (not confirmed): interaction between `hf`'s internal chunked-write path and BeeGFS's write semantics. The extra bytes look like duplicate range overlap from a resumed request, but the math doesn't perfectly fit a clean double-fetch. Worth filing upstream if it reproduces in a minimal repro, but not blocking.

**Workaround that worked:** `git clone https://huggingface.co/Qwen/Qwen2.5-0.5B-Instruct` for the small files (tokenizer, config, vocab, merges) and `curl -L` for the .safetensors. Bypasses the `hf` CLI entirely.

#### `srun --jobid=X --pty bash` hangs (with or without `--overlap`)

To open a second shell on the same allocated node, the canonical trick is `srun --jobid=$JOBID --pty /bin/bash`. This hung indefinitely. Adding `--overlap` (required in Slurm 21+ for stepless overlap into an existing alloc) didn't fix it. Tried `--overlap --mem=0 --cpus-per-task=1 --pty bash` — also hung.

Job state via `scontrol show job` confirmed allocation was healthy (RUNNING, CPUs/Task=8 all used by the original interactive shell). Likely the new step's resource request was waiting on something the first step had locked, and `--overlap` semantics weren't enough.

**Workaround that worked:** plain `ssh n014` from the login node. Direct SSH to compute nodes is allowed (at least when you have an active allocation there). This is also the path that the SSH tunnel from the laptop will go through, so it's actually closer to the eventual production access pattern. I'd been overengineering by routing through Slurm.

For the next session: stop reaching for `srun --jobid` reflexively. If I have an alloc on n014, `ssh n014` is the right second-shell tool.

#### SSH tunnel from laptop didn't bind locally

Tried `ssh -L 11434:n014:28811 ex3` from the laptop. First attempt failed with `bind [127.0.0.1]:11434: Address already in use` — local Ollama install is squatting that port (Ollama default = 11434). Tried again with `ssh -L 11500:n014:28811 ex3`, no bind error this time, ex3 banner came up cleanly. But `curl http://localhost:11500/v1/models` from a fresh local terminal got `Connection refused`.

This is where I tapped out for the day. The ssh session showed the banner without an error message yet localhost wasn't actually serving the forwarded port. Plausible causes:

- The original ssh (the failed-bind one) had set up a ControlMaster/multiplex connection; the second ssh reused that mux and silently dropped the `-L` flag.
- Or the second ssh exited immediately after the banner (terminal still showed banner from connection time but session was already dead).
- Or there's a `Match` rule in my local ssh config that strips port-forwarding for `ex3` somehow.

**Next session diagnostic:** `lsof -iTCP:11500 -sTCP:LISTEN` on the laptop to see if the port is bound at all, then `ssh -v -L 11500:n014:28811 ex3` for verbose handshake. Also `cat ~/.ssh/config | grep -A 20 ex3` to check for ControlMaster lines.

If ControlMaster is the issue, the cleanest fix is `ssh -o ControlMaster=no -L ...` or kill the mux socket before re-running.

### Shutdown

Cleanly stopped at end of session:
- Ctrl+C in the vLLM process on n014 → exit srun shell → allocation released. `squeue -u $USER` showed nothing after.
- `exit` out of both SSH sessions to ex3.
- Killed the tunnel ssh on the laptop.

Nothing burning resources. Model + venv persist on BeeGFS for next time.
