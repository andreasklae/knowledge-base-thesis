# 2026-05-18 — Chess experiment Phase 1 scaffold

Implemented the first working scaffold for [[experiment-chess]] under `experiments/chess/`. This is still a testing-stage implementation, not the agent-learning loop: it supports human and Maia players, allows either side to be configured as either player type, and provides a frontend board for manual testing.

## What was built

- `experiments/chess/backend/` — FastAPI + python-chess backend with a `Player` abstraction, `HumanPlayer`, `MaiaPlayer`, legal-move validation, SSE state streaming, and one backend-owned current game.
- `experiments/chess/frontend/` — React + Vite + chessground interface with white/black player selectors, Maia Elo picker, board interaction, status panel, FEN, and SAN move list.
- `experiments/chess/diary/` — local experiment diary folder for implementation notes that can later be summarized into this knowledge base.

## Important implementation decisions

The backend is the source of truth for game state. Initially the frontend held the created `game_id`, which meant refreshing the page lost the displayed game even though the backend still had it. We corrected the model: the backend now owns exactly one current in-memory game, `POST /api/games` creates or replaces it, and the frontend reloads it with `GET /api/game` after refresh. This keeps the testing UI aligned with the methodological decision from [[diary/experiment-chess/2026-05-16]] that orchestration belongs in the backend.

The chessground board had a subtle integration issue: it was initialized before a game existed with `viewOnly: true`, and chessground does not update `viewOnly` through `api.set()`. The fix was to keep `viewOnly: false` and control interactivity through `movable`, `selectable`, `draggable`, and backend-supplied legal moves. This is a useful small example of a tool integration constraint shaping the workspace design.

## Maia setup

Installed Lc0 locally through Homebrew and downloaded Maia-1 weights for Elo 1100-1900 into `experiments/chess/backend/engines/maia/weights/`. The backend now reports Lc0 as available, and Maia game creation can validate against local weights rather than failing at setup.

## Validation

- Backend: `uv run pytest` — 14 tests passed.
- Frontend: `npm run build` — passed.
- Manual testing: human-as-white vs Maia-as-black can be created; piece movement works after the chessground fix; frontend refresh reloads the backend-owned current game as long as the backend process remains alive.

## Deferred

- Persistent game logs (`games.csv`, PGN, JSON traces).
- Agent player integration.
- Agent skill library and parallel chess wiki.
- Formal progress tracker under `experiments/chess/progress/`.
- Plateau criterion for Phase 1.

The next technical step is to add persistent game export and a minimal game reset/new-game workflow that preserves the singleton backend model while making manual testing smoother.

## 2026-05-18 (second session) — Game lobby + JSON persistence

Completed the deferred game-persistence and multi-game workflow items from the first session.

### What was built

**Backend additions:**

- `app/persistence.py` — new module owning all disk I/O. One JSON file per game at `games/<game_id>.json`. FEN is not stored; it is always re-derived by replaying `uci_moves` through python-chess. Writes are atomic (write to `.tmp`, then `os.replace`). All functions accept `games_dir: Path` as an explicit parameter — no module-level global state — so tests can pass `tmp_path` and remain isolated.
- `app/schemas.py` — added `GameSummary` (game_id, white, black, status, result, turn, move_count, last_move_san, created_at).
- `app/game_service.py` — `Game` dataclass gains `created_at: str`; `_push_move` now calls `_persist` after every move so games survive backend restarts mid-game. New public methods: `load_game` (replay moves from disk, swap `self._game`, restart bot task if active), `list_summaries` (scan all JSON files), `delete` (remove from disk + clear in-memory pointer if it matches). On startup, `GameService.__init__` calls `_try_load_most_recent` so the last active game is automatically restored.
- `app/main.py` — three new endpoints: `GET /api/games` (list), `POST /api/games/{id}/load`, `DELETE /api/games/{id}` (204).
- Tests updated to pass `tmp_path` to `build_test_service` so tests are hermetic. Three new tests added (list, delete, load). All 17 tests pass.

**Frontend additions:**

- `react-router-dom` added. Two routes: `/` → `LobbyPage`, `/games/:gameId` → `BoardPage`.
- `LobbyPage.tsx` — landing screen. Lists all persisted games with players, status (active/finished + whose turn or result), move count, last SAN move, and creation date. Inline collapsible new-game form using the `PlayerSelector` component. Open and Delete buttons per row; delete requires `window.confirm`.
- `BoardPage.tsx` — the existing board + status panel, extracted from `App.tsx`. Calls `loadGame(gameId)` on mount so the correct game is active in the backend even on direct URL navigation or browser refresh. Back link to lobby.
- `App.tsx` replaced with a thin `Routes` shell.

### Key design choice: `games_dir` as explicit parameter

The initial implementation used a module-level `_GAMES_DIR` global set via `init_persistence()`. This caused test pollution: the global persisted across test instances and pointed at the real `games/` directory. The fix was to make every persistence function accept `games_dir: Path` as its first argument and have `GameService` store it in `self._games_dir`. Tests pass `tmp_path`; production passes `settings.games_dir`. No global state.

## 2026-05-18 (third session) — Bug fix + game-over display

Two small but important changes made during manual testing.

### Bug fix: DELETE returned empty body, frontend crashed

The `DELETE /api/games/{id}` endpoint correctly returns HTTP 204 No Content. The backend was correct. The bug was in the frontend `request<T>()` helper in `api.ts`: it unconditionally called `response.json()` regardless of status code, causing `"Unexpected end of JSON input"` on any 204 response. Fix: added an early return for 204 before the `.json()` call.

```typescript
if (response.status === 204) {
  return undefined as T;
}
return response.json() as Promise<T>;
```

This is a pattern-level fix: any future endpoint returning 204 will work without needing its own special handling.

### Feature: game-over banner and termination reason

When a game ends the board page previously showed nothing distinctive — result appeared as plain text in the status panel's "Result" row, easy to miss.

**Backend change:** Added `termination: str | None` to `GameState` in `schemas.py`. Populated from `board.outcome(claim_draw=True).termination.name` in `Game.state()` in `game_service.py`. Uses python-chess's `Outcome.termination` enum, which covers: `CHECKMATE`, `STALEMATE`, `INSUFFICIENT_MATERIAL`, `SEVENTYFIVE_MOVES`, `FIVEFOLD_REPETITION`, `FIFTY_MOVES`, `THREEFOLD_REPETITION`. Field is `None` while the game is active.

**Frontend changes:**

- `GameState` interface in `types.ts` gains `termination: string | null`.
- `BoardPage.tsx` — added `describeOutcome(game)` which maps `(result, termination)` to a human-readable headline and detail string, and a `kind` discriminant (`'win'` | `'draw'`) for styling. Examples:
  - `CHECKMATE` + `1-0` → "White wins by checkmate" / `1-0`
  - `STALEMATE` → "Draw by stalemate" / `½-½`
  - `INSUFFICIENT_MATERIAL` → "Draw by insufficient material"
  - `1-0` with unknown termination → "White wins" / `1-0`
- A `.result-banner` div is rendered between the nav panel and the board layout when `game.status === 'finished'`. It is visually prominent — full-width, coloured background (green for a decisive result, warm cream for a draw), large headline, smaller detail.
- The "Result" row in the status panel also now shows the termination reason in parentheses (e.g. `1-0 (checkmate)`).
- CSS: `.result-banner`, `.result-banner.win`, `.result-banner.draw`, `.result-headline`, `.result-detail` added to `styles.css`.

### Reflection

The 204 bug is a good example of a tool-integration failure mode in web development: the `request` helper was designed for JSON endpoints and made an implicit assumption (all responses have a body) that the DELETE endpoint violated. The fix required no interface change — just a guard on the status code inside the single shared utility. This kind of boundary-condition gap is exactly what tests should catch; worth noting that none of the existing tests covered the actual HTTP DELETE path through the frontend API client (tests only covered the backend).
