# 2026-05-24 — Chess backend switched to self-hosted Gemma on eX3

While bringing up the eX3 inference server today (see [[diary/module-llm-server/2026-05-24]]) I noticed the chess experiment's `AgentPlayer` was hard-wired to `AzureProvider`. Even though the skillful-agent SDK now supports `SKILL_AGENT_EX3_BASE_URL`, the chess backend builds its own `Agent` instance in `_build_agent` rather than calling the skillful-agent HTTP server — so the new env var wasn't being honoured.

## What changed

- **`experiments/chess/backend/app/agent_player.py`**: `_build_agent` now branches on `SKILL_AGENT_EX3_BASE_URL` (highest priority) → `SKILL_AGENT_AZURE_ENDPOINT` → OpenAI public API. This mirrors the priority order in `skillful-agent/server/app.py:_build_agent`.
- **`experiments/chess/backend/.env`**: switched to eX3 by default. Default model is now `google/gemma-4-31B-it`. Azure lines kept but commented out.

## Implication for the experiment

The inference backend is now part of the env, not the repo. `skill_repo_sha` in the CSV will not change when the backend toggles, but `model` will. Whoever reads the data later needs both columns to reconstruct the agent configuration.

For Phase 1 calibration runs going forward, the agent will be playing on Gemma 4 31B-it served from eX3 unless the env is flipped back to Azure. This is a deliberate move — staying on a self-hosted model means:
- No quota or cost surprises mid-batch.
- Sampling temperature, max tokens, and any future model swap are fully under our control.
- The experiment is reproducible by anyone with eX3 access.

The risk is the opposite of the Azure path: eX3 nodes can drop, the SSH tunnel can die, and a Slurm allocation has a time limit (typically 4 hours). Long Phase 1 batches will need either tunnel watchdogs or a way to resume mid-batch. Flagged for the batch runner's next iteration.

## Open todo for chess

- Decide whether the production Phase 1 runs target Gemma 4 31B-it on eX3 or fall back to GPT-4o-mini on Azure. The former is what we're running with today by default; the decision should land in a decision record before the first calibration batch.

## Addendum — same day, later session

Three follow-on decisions landed during the same day as the model switch above. They settle the methodology questions that came out of running the first calibration attempts on Gemma:

1. **Initial agent ELO set to 600**, not the conventional 1200 — informed prior from observed gameplay. Reduces calibration-waste games. See [[decisions/2026-05-24-initial-elo-600]].
2. **Reasoning must precede the move.** A direct vLLM SSE probe confirmed Gemma's default is tool-first: it called `get_weather` immediately with no preamble, then in a separate experiment with explicit "reason step by step" instructions, streamed 34 seconds of reasoning before the tool call. The chess system prompt now enforces an ordered sequence (`use_skill` → `list_legal_moves` → 2–4 sentences of reasoning → `make_move` → end turn). The frontend labels text after `make_move.py` as `post-move` and renders it muted to signal post-hoc commentary. See [[decisions/2026-05-24-reason-before-move]].
3. **Per-turn fresh context.** One calibration game died with a 400 "maximum context length is 65536 tokens" — the accumulated conversation hit Gemma's window. Baseline runs now call `clear_conversation()` at the start of every turn; the agent sees system prompt + skill list + one user message (opponent move + FEN). The FEN encodes complete state; cross-turn memory is reserved for future configurations to measure against. Aborted games are recorded with `result=""` and a new `aborted_reason` CSV column; ELO is unchanged for them. `skill_agent` gained a typed `AgentContextOverflowError`. See [[decisions/2026-05-24-per-turn-fresh-context]].

Combined, the configuration is now methodologically clean enough that the first real calibration batch (Maia or chesscom, Gemma 4 31B-it, initial ELO 600) can produce the Phase 1 baseline number. The "production backend" open question above is settled in favour of Gemma; the ADRs are the durable record.

4. **Ranked vs experimental logging.** A fourth decision landed: every agent game is written to `ranked.csv` (when on clean `main`, also updates ELO) or `experimental.csv` (any other git state, ELO frozen). All game data is now tracked in git so `git revert` on a calibration PR reverses both code and CSV rows together. PRs are the unit of "agent configuration version"; closed-without-merge PRs become research records of what was tried and rejected. Frontend banner shows the live mode. See [[decisions/2026-05-24-ranked-vs-experimental]].
