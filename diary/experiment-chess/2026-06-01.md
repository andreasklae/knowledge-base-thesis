# 2026-06-01 — pydantic-ai profile mismatch fix + KB documentation

## What happened

Investigating whether tool calls were being handled optimally for Gemma 4
via vLLM. The short answer was no — there were two pydantic-ai profile
defaults that were actively wrong for this stack and were causing
intermittent failures we had attributed to model unreliability.

## The bug

pydantic-ai's `OpenAIProvider` uses an `OpenAIModelProfile` with two
defaults that don't apply to vLLM/Gemma:

**`openai_supports_strict_tool_definition=True`** — causes pydantic-ai to
include `"strict": true` inside every tool's `function` definition in the
JSON payload. Gemma's chat template renders tool definitions in a completely
different format (brace-key syntax, uppercase types, `<|tool>...<tool|>`
delimiters — not JSON at all). vLLM's template ignores `strict` during
rendering, but vLLM itself may reject or mishandle the field depending on
version. This was a known contributor to the intermittent HTTP 400
"malformed JSON" errors we'd been seeing.

**`openai_supports_tool_choice_required=True`** — when `allow_text_output=False`,
causes pydantic-ai to send `tool_choice: "required"`. Behavior for this is
undefined in vLLM/Gemma. Since we start vLLM with `--enable-auto-tool-choice`,
sending `tool_choice: "auto"` (what you get with `False`) is the correct path.

Neither flag was on the radar because both are buried in pydantic-ai defaults;
we were using the provider as if it were pointed at OpenAI.

## The fix

```python
from pydantic_ai.models.openai import OpenAIChatModel, OpenAIModelProfile

profile = OpenAIModelProfile(
    openai_supports_strict_tool_definition=False,
    openai_supports_tool_choice_required=False,
)
model = OpenAIChatModel(model_name, provider=provider, profile=profile)
```

Applied in two places:
- `software/skillful-agent/server/app.py:_build_agent` (ex3 branch only)
- `experiments/chess/backend/app/agent_player.py:_build_agent` (ex3 branch only)

Azure and OpenAI public API paths use `profile=None` (pydantic-ai defaults),
which is correct for those providers.

## Broader context: Gemma's native tool format

While investigating, I wrote up a full concept page on how the three-layer
stack (pydantic-ai → vLLM → Gemma 4) actually fits together. Key things
that weren't obvious before this session:

- Gemma 4 uses a completely non-JSON native tool format. vLLM translates
  between OpenAI wire format and Gemma's format invisibly — pydantic-ai
  never sees the native tokens.
- The `--tool-call-parser gemma4` vLLM flag is mandatory. The `pythonic`
  parser (which looks like the obvious default) causes Gemma's tool-call
  tokens to leak into the `content` field as raw text instead of being
  parsed into `tool_calls`.
- Thinking tokens: Gemma wraps chain-of-thought in `<|channel>thought...<channel|>`.
  vLLM's chat template strips this from history automatically via a
  `strip_thinking` macro. We must NOT set `openai_chat_thinking_field` on
  the profile — doing so routes thinking through a separate field that
  bypasses the template's stripping, which would violate Gemma's multi-turn
  rule (prior turn thoughts must not appear in history).
- Disabled thinking still emits an empty channel block. This is by design.

See [[gemma4-vllm-pydantic-ai-integration]] for the full write-up.

## Relationship to the 05-28 open question

The diary entry from 2026-05-28 noted: "Open structural question (deferred):
whether `skillful-agent` is fighting Gemma 4 (malformed tool-call JSON,
chain-of-thought channel breakage)." Today's fix addresses one layer of
that: the `strict` field in tool definitions was a concrete cause of the
HTTP 400 failures. The channel breakage was already handled correctly —
vLLM's template does the right thing; we just confirmed it.

The `guided_json` / constrained-decoding question remains deferred. The
profile fix removes a source of noise; if empty-args loops or malformed
calls persist in the next batch, that's when to revisit.
