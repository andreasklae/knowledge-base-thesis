# 2026-05-20 — End-to-end chess.com opponent: lifecycle overhaul, DOM rediscovery, first agent vs chess.com game

## Summary

Today the chess.com opponent went from "wired in the backend, never tested" (2026-05-19) to a working end-to-end pipeline: the agent (white) plays a chess.com Engine bot (black) through a Playwright-driven Chrome window, and the result shows in the local app's board UI. Chess.com checkmated the agent in the first real game. Most of the day was spent on lifecycle and infrastructure problems — not on the agent itself.

The session covered five distinct problem areas:

1. **UI wiring** so the chess.com player is selectable.
2. **Frontend rendering bugs** that made the board collapse to a thin strip — the chessground library's absolute-positioned children don't give their parent a height, so the wrapper had to be sized explicitly via CSS container queries.
3. **Backend startup and Maia silent failure** — Maia engine processes died silently because the lc0 path wasn't set and exceptions weren't propagated.
4. **Browser lifecycle redesign** — the persistent Chrome user-data dir kept leaking `SingletonLock` files between games, so every second game opened a stale tab and failed. Switched to ephemeral temp profiles and an eager-launch lifecycle.
5. **chess.com DOM rediscovery** — the live page has changed since the driver was written; the Engine group is now at the bottom of a long scrollable list of themed bot groups, and two pre-game overlays (intro modal, cookie banner) intercept clicks.

## What works at the end of the day

- Lobby has constraints baked in: chess.com is only selectable as black; agent is only selectable as white. ELO for chess.com is a dropdown of the 25 actual slider ratings (250–3200), not a free-form input.
- "Start game" against a chess.com opponent eagerly launches Chrome before the HTTP response returns, so the frontend either gets a working game state or a clean 502 with a real error message. No silent timeouts after navigation.
- The driver dismisses both the chess.com first-run "Play the bots" intro modal and the OneTrust cookie banner before interacting with the bot picker, scrolls the Engine group into view, expands it, sets the slider via React-compatible native setter, and clicks Play.
- The game ends → all players' `close()` methods are awaited → Chrome quits → the temp profile is deleted. Starting another game from a fresh profile works on the first try.
- Active chess.com games viewed from the lobby are now served from the in-memory `_game` rather than reloaded — chess.com games cannot be resumed because their browser session is ephemeral, but the active game keeps working as long as the process is alive.
- An agent vs chess.com game ran to completion with the agent receiving prompts of the form "Opponent played c5. Current position (FEN): ..." and chess.com replying through pointer-event-simulated moves on the live board.

## Detailed log

### 1. Lobby UI wiring (small, clean)

`/api/player-types` was already returning entries for `human`, `maia`, `agent` — but not `chesscom`, even though `PlayerFactory` and `PlayerConfig` had the type. Added `PlayerTypeInfo(type="chesscom", elo_required=True, allowed_elos=list(CHESSCOM_ELOS))`. Frontend `PlayerSelector` got a `chesscom` branch with an ELO `<select>` mirroring the Maia pattern; `cleanConfig()` and `formatPlayer()` in `LobbyPage.tsx` now know about the new type.

The ELOs are the canonical 25 from `chesscom_driver.mapping.POSITION_TO_RATING`, duplicated as a `CHESSCOM_ELOS` tuple in `schemas.py` so validation doesn't have to import the driver package. The schema rejects ELOs outside that set.

### 2. Frontend board rendering

The chessground board was rendering as a thin vertical strip — only the coordinates were visible, no squares. Two compounding issues:

**Issue A: `.cg-wrap` had no height.** Chessground's internal DOM positions all its children with `position: absolute`, so they contribute zero height to the parent. `aspect-ratio: 1` on the wrapper does nothing here because there's no intrinsic content height to maintain a ratio against.

**Issue B: `100vw` was the viewport width.** First attempt at fixing the height used `height: min(80vh, 100vw)` — but `100vw` is the *viewport* width, far larger than the grid column the board lives in. Result: the board was as tall as the viewport but only as wide as the column → tall thin strip.

**Fix:** CSS container queries. `.board-panel` gets `container-type: inline-size`, then `.board-frame` uses `width: min(80vh, 100cqw); height: min(80vh, 100cqw)`. `100cqw` is the container's inline size (= column width), so width and height are equal and constrained by both the column and the viewport height. The board is now perfectly square inside the grid column.

`ChessBoard.tsx` also got a `ResizeObserver` in its mount effect that calls `apiRef.current?.redrawAll()` whenever the element resizes. Chessground caches pixel geometry on first mount; if CSS layout hasn't resolved yet (which is common with grid + container queries), the cached geometry is wrong and pieces render at offset positions. The observer guarantees a redraw after layout settles.

### 3. Backend startup and the silent Maia failure

Several non-obvious failures cascaded here.

**Conda was auto-activating its base env in the shell**, which had a different Python than the project's uv-managed `.venv`. `uvicorn app.main:app` would hang because the wrong interpreter couldn't find the project's dependencies. Fixed via `conda config --set auto_activate_base false`. The user later flagged this as a drifting fix — the symptom was suspicious but the cause was outside the project.

**`pydantic_settings.sources.providers` was missing** after a partial install. Resolved by `rm -rf .venv && uv sync` — a hard reset of the venv. Not a code change, just an environment reset.

**Maia (lc0) silently never moved.** The root cause was three-layered:

1. `CHESS_LC0_PATH` was not set in `backend/.env`. lc0 lives at `/opt/homebrew/bin/lc0`, which is on the user's shell PATH but not on the FastAPI process's PATH. `MaiaEngineRunner.validate_available()` therefore raised a `PlayerError` saying lc0 wasn't found.
2. The `_run_until_human_or_finished` bot loop caught only `PlayerError`. Any other exception (including the `FileNotFoundError` from a direct `subprocess` call, or a malformed weights file) propagated unhandled, killing the asyncio task. No publish to the SSE channel, no error message in the UI — Maia just never moved.
3. The downloaded Maia weights were ~1.2 MB instead of the expected ~80 MB. The official URL is `https://github.com/CSSLab/maia-chess/releases/download/v1.0/maia-{elo}.pb.gz`; the original download had silently truncated.

**Fix:** `CHESS_LC0_PATH=/opt/homebrew/bin/lc0` in `backend/.env`; broadened the bot-loop catch to `except Exception`; re-downloaded the weights. The bot-loop catch broadening means any future engine failure produces a visible error event on the board SSE channel rather than a dead task.

Also fixed two unrelated signature mismatches: `HumanPlayer.get_move()` and `MaiaPlayer.get_move()` didn't accept the `last_san: str | None = None` keyword that the `Player` ABC was updated to require on 2026-05-19. The agent path worked because `AgentPlayer` did accept it; Maia hit the `unexpected keyword argument 'last_san'` error as soon as the bot loop tried to call it. Both signatures are now ABC-compliant.

### 4. Browser lifecycle redesign

This was the largest piece of work today. The original design had `ChessComPlayer.get_move()` lazily launching the browser on the first move (described in [[diary/experiment-chess/2026-05-19]] as "lazy launch is the right design here"). In practice this broke as soon as a second game was attempted:

- Persistent profile: the driver used `backend/chesscom-profile/` as the Chrome user-data dir. Chrome writes a `SingletonLock` (plus `SingletonSocket`, `SingletonCookie`) file when it owns the profile, and these files are not always cleaned up — especially if the process exits abnormally or if a stale browser is still alive.
- Second-game launch: Playwright tries to open a new persistent context, sees the lock, and exits with `"Opening in existing browser session"` followed by `Error: kill EPERM`. The visible symptom was a new about:blank tab opening with no game.
- No close on game over: `Game` dataclass had no `close_players()` method. When a game finished or was replaced, the browser stayed open indefinitely.

**Redesign decisions:**

| Decision | Rationale |
|---|---|
| Ephemeral temp profile per game (`tempfile.mkdtemp(prefix="chesscom-")`) | Guarantees a clean state every game. No SingletonLock collisions. Trade-off: chess.com login is not preserved — but anonymous play on `/play/computer` works fine for Engine bots, so this is acceptable. |
| Eager browser launch in `create_game()` (not lazy in `get_move()`) | The frontend sees a clean 200 (game ready) or a clean 502 (Chrome couldn't launch) on `POST /api/games`. No mid-game "browser couldn't open" surprises. The lobby shows "Launching chess.com browser… this can take 10–20s" while the request is in flight. |
| Restrict chess.com to black-only and agent to white-only at the schema layer | `CreateGameRequest.validate_sides` raises if `white.type == "chesscom"` or `black.type == "agent"`. Frontend `PlayerSelector` filters the available options per colour. Two enforcement points, one source of truth. |
| ELO dropdown of the 25 known ratings | The previous free-form 250–3200 input lied to the user — the driver's `closest_position()` would silently round to one of 25 values. A dropdown makes the actual coverage visible. |
| Awaited cleanup at every transition | New `Game.close_players()` is awaited in `create_game` (closing the prior game before launching the new one), `load_game` (same), `delete`, game-over detection in the bot loop, and the FastAPI lifespan shutdown. No more fire-and-forget `asyncio.create_task(close())` racing with new launches. |
| `ChessComPlayer` subclass owns the temp dir and wipes it on close | Encapsulates the lifecycle: `__init__` mkdtemp, override `close()` to `shutil.rmtree(self._tempdir, ignore_errors=True)` after super().close(). |

The driver package gained an `async def start(self)` method that runs `launch() + start_game()`; the host calls it via `Game.start_players()` which iterates players and calls `start()` on any that have it. `MaiaPlayer` and `HumanPlayer` don't have `start()` so they're no-ops. If `start_players()` raises, `create_game` immediately awaits `close_players()` and re-raises as a 502 — so a chess.com launch failure cleans up the temp dir and doesn't leak.

`_try_load_most_recent()` was removed entirely. It had been auto-loading the most recent game at startup, which meant the backend instantiated agent players (and potentially chess.com browsers) before the user even visited the app. The lobby already lists all games; users open the one they want explicitly. No agent or chess.com process exists until a game is explicitly created or loaded.

The SSE endpoints (`/api/game/events`, `/api/games/{id}/events`, `/api/games/{id}/agent-events`) were producing a noisy `"Caught handled exception, but response already started"` traceback in the logs when there was no active game in memory. The cause: the subscribe call lived inside the `stream()` generator, so the 404 fired *after* StreamingResponse had already started sending headers. Moved the `subscribe_*` calls out of the generator, so any 404 fires cleanly before the response begins.

### 5. chess.com DOM rediscovery

The biggest single surprise. The driver was built against a chess.com `/play/computer` page that had:

- An "Engine" accordion group identifiable by `[data-cy="bot-group-Engine"]`.
- A 25-step slider `input.slider-input` inside it.

The current live page has:

- Twelve themed bot groups: Pirates, Beginner, Intermediate, Advanced, Master, Adaptive, Athletes, Musicians, "Chess the Musical", Creators, Top Players, Personalities — and then "Engine" at the bottom.
- The Engine group is collapsed by default. Its toggle is at the bottom of a long scrollable list inside the sidebar.
- A first-run intro modal (`dialog[data-cy="intro-modal"]`) with a "Start" button gates everything until dismissed.
- A OneTrust cookie banner (`#onetrust-consent-sdk`) that intercepts pointer events on whatever sits below it — including the Engine toggle once it's scrolled into view.

The driver was failing without a clear error message: clicks on the Engine toggle were being intercepted by the OneTrust banner. The user provided the rendered HTML, which showed the intro modal on top, the cookie banner below, and the Engine group at the very bottom of the bot list.

**Diagnostic step that paid off:** wrote a one-off `/tmp/chesscom_probe.py` script that drove a headed Chrome through the full flow with verbose logging at each step. The probe established the correct order: dismiss intro modal first (the modal sits on top of the cookie banner, so cookie clicks fail until the modal is gone), then hide the cookie banner via JS (`document.getElementById('onetrust-consent-sdk').style.display = 'none'` — clicking the Accept button is unreliable across locales, but hiding the container is enough to release pointer events), then scroll the Engine toggle into view, expand it, drive the slider, click Play. The probe ran clean end-to-end and printed `board ready? True` with the starting FEN.

The driver was then updated to follow exactly that order. Key changes to `chesscom-driver/chesscom_driver/driver.py`:

- `_dismiss_first_run_modal()` — clicks "Start" on the intro modal first, then hides the OneTrust container via JS. Both wrapped in try/except so a clean session (no overlay) skips silently.
- `_ensure_engine_group_open()` — uses Playwright locators scoped to `[data-cy="bot-group-Engine"]`, calls `scroll_into_view_if_needed()` before clicking the toggle, and waits for the slider to appear *inside that specific group* rather than anywhere on the page (other sliders may exist).
- `_set_slider_position()` — selector is now scoped to `[data-cy="bot-group-Engine"] input.slider-input` so it can't accidentally drive a slider in a different group.

The probe script lives at `/tmp/chesscom_probe.py` and is useful to keep in mind as a debugging pattern: when the driver fails inscrutably against a live page, run a verbose probe with `has(selector)` checks at each step and decide based on what the page actually shows, not what the old code assumed.

### 6. Lobby/board page navigation

A late-session bug: opening an in-progress chess.com game from the lobby triggered `POST /api/games/{id}/load`, which calls `load_game()`. `load_game()` now refuses to resume any active chess.com game because the browser session is ephemeral and gone. So agent-vs-chesscom games that were running in the backend were unviewable from the frontend — "Active chess.com games cannot be resumed — start a new game."

**Fix:** the BoardPage now tries `GET /api/games/{id}` first. If the game is already the active in-memory `_game`, GET returns 200 and the page renders. Only if GET fails (i.e. the requested game isn't the current one in memory) does the page fall back to `POST /api/games/{id}/load`. The lobby's "Open" button no longer pre-loads — it just navigates and lets BoardPage handle the get-vs-load decision.

This also fixes a subtle issue with finished agent games: there's no reason to re-instantiate the agent's pydantic-ai model just to view a completed game's board state and history. GET is sufficient.

## What broke and how to find it

- **Chrome reuses profiles silently.** A leftover SingletonLock in `chesscom-profile/` was the symptom of every "Opening in existing browser session" error. With temp profiles, this class of bug is gone — but if it returns under a future design, check `~/<profile>/SingletonLock` first.
- **Maia errors that aren't `PlayerError` will kill the bot loop.** The catch is now `except Exception`, but watch for any narrowing of that catch in future refactors. Engine subprocesses can fail in many ways (binary missing, weights corrupt, OOM, segfault).
- **Chessground needs explicit height and a post-mount redraw.** The `ResizeObserver` is load-bearing — without it, the board renders at zero size whenever CSS layout takes more than a tick to settle.
- **chess.com's `/play/computer` page DOM is not stable.** This is the second time in two sessions the driver has needed adjustment. The maintenance pattern is: when something stops working, write a probe script, walk through the flow with verbose logging, and update selectors based on what the live page actually shows.

## Methodological note

Today's work is almost entirely infrastructure: lifecycle, UI, DOM scraping, error visibility. Zero changes to the agent itself. This is exactly the pattern the thesis predicts — see [[tool-fairness]] and [[deterministic-tools-hypothesis]]: making an opponent available to the agent is server-side infrastructure work, equivalent to wrapping Stockfish or Maia. The agent's prompt is unchanged ("Opponent played c5. Current position (FEN): ..."), the skill scripts are unchanged, the model is unchanged. The agent has no awareness that one of its opponents now runs through a Playwright-controlled browser instead of an lc0 subprocess. This is the right separation: opponent provisioning is a host concern, not an agent concern.

The fact that "wire a chess.com bot as an opponent" required dismissing two HTML overlays, scrolling a sidebar, and rediscovering selectors against a moving target is also a real datapoint for the broader thesis claim that *agent infrastructure is the bottleneck*. The skill (`chess-player`), the agent, and the model could all be perfect — and the experiment would still be blocked on whether a temp Chrome profile can find the Engine slider on a redesigned chess.com page. Infrastructure work like today's is the work, not the prelude to the work.

## Afternoon: ELO tracking, batch runner, evaluation needle, and edge-case alignment

After the first agent-vs-chess.com game completed in the morning, the rest of the day was spent making the experiment runnable — turning "one game at a time, by hand" into "a batch of N games where the agent climbs an Elo ladder unattended." A full decision record covering the methodology choices lives at [[decisions/2026-05-20-elo-and-batch-runner]].

### Repository state

The chess experiment got its own git repo today via `git init` in `experiments/chess/`. Every change to the agent — skills, prompts, model config, driver code — is now committed there, and the HEAD SHA is what we stamp into `games.csv.skill_repo_sha`. The `.gitignore` was set up to keep secrets (`.env`), the venv, Maia weights (11 MB of `.pb.gz` binaries), and the legacy `chesscom-profile/` out of the repo while keeping `games/` and `batches/` tracked as experiment artefacts. Game state files are part of the thesis record.

### Edge cases settled before the batch runner

Two correctness issues from the morning's session were resolved before building the batch infrastructure on top of them:

**Underpromotion synchronization.** chess.com's driver has `autoPromote: true` so the click sequence for a promotion auto-queens regardless of the UCI suffix. If the agent decided to underpromote (=N, =R, =B), the host would record the underpromotion while chess.com played a queen — the two boards would silently diverge from that move onward. Decision: at sub-1100 ELO, underpromotion against weak bots is almost never the strongest move, and the chess.com driver only exists to cover sub-1100. So the fix is to **force-queen** at the host's `_push_uci_move` boundary when the opponent is chesscom: any move with `promotion != QUEEN` is rewritten to `QUEEN` before legality check and pushed onto the host board. Three new lines. Host and chess.com stay aligned; the agent's reasoning trace still records what it picked. From Maia 1100 upward this code path never fires.

**Draw-rule synchronization.** python-chess's `is_game_over(claim_draw=True)` reports threefold repetition and 50-move rule as game-over because the player *can* claim them. chess.com does not auto-claim these — both are claimable draws, and chess.com will keep playing until something mandatory (5-fold, 75-move, stalemate, insufficient material, checkmate) happens. So if the host called those games over while chess.com kept playing, the bot loop would terminate while the browser session was still live. Decision: `Game._claim_draw` is `False` for any game with a chess.com player, `True` for all others. Game-over is now driven by mandatory termination only when chess.com is involved. The `claim_draw` flag also threads into `Game.outcome()`, `Game.result()`, and `Game.is_over()` consistently. The same change generalised the API: `is_over()` is now the canonical "is this game done?" check, replacing scattered `board.is_game_over(claim_draw=True)` calls throughout `game_service.py`.

### ELO infrastructure

`app/elo.py` is the new module owning everything ELO. It exports:

- `expected_score(own, opponent)` — standard logistic with 400-point scale.
- `update_elo(own, opponent, result)` — `own + K * (score - expected)` with `K=32`.
- `pick_opponent_elo(agent_elo, last_result, pool)` — the win-up / loss-down / draw-down opponent selector. Returns one of the 9 Maia ratings or 25 chess.com ratings.
- `AgentEloState` — small persistent dataclass holding the agent's current ELO, games-played counter, and last result. Lives at `<backend>/games/agent_elo.json`. Starts at 1200.
- `parse_game_result(result_string, agent_color)` — translates "1-0"/"0-1"/"1/2-1/2" into "win"/"loss"/"draw" from the agent's perspective. Returns None for unfinished games.

The module is fully unit-testable on its own — no FastAPI, no game state, no I/O beyond the optional `AgentEloState.save/load`. I verified the math by inspection: `update(1200, 1200, win) = 1216`, `update(1200, 1200, loss) = 1184`, `update(1200, 1200, draw) = 1200`. Expected score `(1500 vs 1200)` ≈ 0.85.

### Stockfish evaluation needle

`brew install stockfish` puts the binary at `/opt/homebrew/bin/stockfish`. python-chess's `chess.engine.SimpleEngine.popen_uci` drives it as a UCI subprocess. `EvalService` in `app/eval_service.py` owns one long-lived engine per process, gated by an `asyncio.Lock` so concurrent eval requests serialise cleanly. `evaluate(board)` runs `engine.analyse(board, Limit(depth=15))` in a worker thread (via `asyncio.to_thread`) and returns `{"cp": int | None, "mate": int | None}` from White's perspective.

If Stockfish is missing or fails to start, the service quietly returns nulls — the UI just renders a neutral bar. No game flow depends on the eval.

Integration: `GameState` gained `eval_cp: int | None` and `eval_mate: int | None`. Every time a move is pushed (either via the bot loop or via human submit), `_update_eval(game)` is scheduled as a background task. It computes the eval, checks the game is still the active one (and the FEN hasn't moved on), then publishes a fresh state event. ~100–300 ms latency per move.

Frontend: `EvalBar.tsx` is a thin vertical bar next to the board. The white-fill fraction is a sigmoid (`2/(1 + exp(-0.004*cp)) - 1`, mapped to [0,1] — the same scaling lichess uses) so mid-range advantages feel meaningful while extremes saturate cleanly. Centipawn label sits at the bottom when white is winning, top when black is. Mate scores collapse the bar fully to one side. The bar's height tracks the board's responsive sizing via the same `min(80vh, 100cqw)` container-query pattern, mirroring CSS architecture set up two days ago.

### Pause/play for any agent game

A pause toggle now appears on the board page whenever an agent (white) is one of the players and the game is active. Backend exposes `POST /api/games/{id}/pause` and `/resume`, which flip `Game.paused`. The bot loop checks `game.paused` at the top of each iteration and returns cleanly if set. Resuming re-schedules the bot loop. Pause is honoured **between turns**, so an in-flight `player.get_move()` (which could include an OpenAI completion mid-stream) runs to completion before the loop actually halts. The UI shows "Paused — current turn finishes, then pauses" so the user understands the lag.

`GameState.paused` is published on the SSE channel so the UI reflects the state immediately.

### Batch runner

The largest piece of today's afternoon. New modules `app/batch_service.py` (state + persistence) and `app/batch_runner.py` (orchestration). The shape:

- **Batch model**: `{batch_id, label, pool, total_games, status, games[], current_game_id}`. Persisted as one JSON file per batch under `<backend>/batches/`.
- **BatchService**: pure persistence + factory. Doesn't know about game loops. Owns `AgentEloState` too.
- **BatchRunner**: subscribes to `GameService.on_game_finished`. When a game finishes:
  1. Determines the agent's result from `game.result()`.
  2. Updates `AgentEloState` via `apply_result()` and saves.
  3. Appends a `GameRecord` to the batch and saves.
  4. Sets `_next_game_event` so the runner's outer loop picks the next game.
- Outer loop: re-loads the batch on each iteration so external `pause`/`stop` calls take effect cleanly. If the batch is paused or stopped, the loop idles or exits. If the previous game finished and we're under `total_games`, it creates the next game.
- Next-game creation: `BatchService.next_create_request(batch, agent_elo)` uses `pick_opponent_elo` to choose the opponent's rating from the batch's pool, builds a `CreateGameRequest` with `white=agent` and `black={pool, elo=opp_elo}`, calls `GameService.create_game()`, then stamps `batch_id`, `batch_name`, and `agent_elo_before` onto the newly-created Game so the CSV row carries them.

**Ordering subtlety:** `_on_game_finished` callback fires *before* `_record_game_end` writes the CSV row. This means the batch runner can compute the new ELO and stamp `agent_elo_after` onto the Game object *before* the CSV captures it. Without this ordering the `elo_after` column would always lag by one row. The change is one block in `_run_until_human_or_finished` — call the callback first, then write CSV.

**Endpoints**: `GET/POST /api/batches`, `GET /api/batches/active`, `POST /api/batches/{id}/start`, `/pause`, `/stop`, plus `GET/POST /api/agent-elo` for inspecting and resetting the persistent ELO state.

### Frontend batch page

New route `/batch` (linked from the lobby header as "Batches →"). Three sections:

- **Agent ELO banner** at the top: big number for current ELO, games-played count, last result, with a Reset button (confirms before resetting to 1200).
- **Active batch card** (if any): batch label, status chip, progress bar (`N/total_games`), current opponent, live link to the current game's board, last-10 results as W/L/D chips, full results table in a collapsible `<details>`. Pause/Resume/Stop buttons.
- **Create form**: label (free text, max 80 chars), pool dropdown (Maia / chess.com), total games (1–500). Submits to `POST /api/batches`.

A `setInterval(1500ms)` poll refreshes the active batch's state — game-level granularity is fine here because the actual games run on a multi-second timescale. The interval will be replaced by SSE in a future polish pass if the polling proves janky.

### CSV schema (final)

`games.csv` columns are now `game_id, batch_id, batch_name, datetime, white_type, black_type, opponent, opponent_elo, result, elo_before, elo_after, skill_repo_sha, model, temperature, agent_log_path`. ## Late afternoon: matchmaking made adaptive, workspace restructure, first commit-ready state

Two follow-ups to the morning's batch infrastructure: a structural cleanup of the Python packaging, and a methodological upgrade to the matchmaking algorithm after the first real batch ran too slowly.

### uv workspace restructure

The original layout had `experiments/chess/backend/` as a standalone uv project and `experiments/chess/chesscom-driver/` as a sibling package referenced from the backend via `{ path = "../chesscom-driver", editable = true }`. That worked initially but produced a `.pth`-based editable install. CPython's `.pth` processor silently dropped the entry because the path contained an en-dash (`Dokumenter – MacBook Air`) and `addsitedir` returned without registering it — every `from chesscom_driver import …` failed at runtime even after `uv sync` reported success.

The structural fix: a **uv workspace**. Created `experiments/chess/pyproject.toml` as a workspace root declaring `backend` and `chesscom-driver` as members. Changed the backend's source declaration to `chesscom-driver = { workspace = true }`. Now uv handles the cross-package install as proper site-packages entries (no `.pth` indirection), the single workspace venv lives at `experiments/chess/.venv/`, and `uv sync --all-packages` from any directory inside the workspace installs everything.

This is also the idiomatic layout for multi-package Python repos and survives renames/moves of the parent directory, which the `.pth` approach didn't.

### Adaptive K-factor and streak-based opponent stepping

The user ran a 10-game Maia batch and stopped it early because calibration felt slow — the agent was losing every game and the next opponent only dropped one Maia rating per game (1300 → 1200 → 1100 → and then stuck at 1100 because there's no lower Maia opponent). This matches the prediction in the decision record (single-step opponent selection has linear calibration), but observing it under real conditions motivated upgrading the algorithm.

Two coupled changes in `app/elo.py`:

**Adaptive K-factor.** `k_factor(games_played)` returns 40 for the first 15 games, then 20. This mirrors FIDE's provisional-player rule and chess.com's calibration window. Higher K during the unknown-skill phase moves the rating faster; lower K once we have a usable estimate keeps subsequent games from oscillating. The CSV doesn't gain a new column — `games_played` is implicit in row order and `elo_before`/`elo_after` let downstream analysis reconstruct the K that was applied.

**Streak-based opponent stepping.** `AgentEloState` gained a `streak` field — a signed integer that increments on wins, decrements on losses, flips sign on a direction change, and halves toward zero on draws. The next opponent's grid position is the agent's anchor index plus a step computed from the streak: `2^(|streak|-1)` capped at `MAX_STEP=4`. So:

- A 1-game streak: step ±1 (same as before).
- 2 wins in a row: step +2.
- 3 wins: step +4.
- 4+ wins: step +4 (capped).
- Same on the loss side, with sign flipped.

For an agent starting at Maia 1200 with true skill ~300, the new algorithm reaches the bottom of the chess.com pool in 3 games (1200 → 1000 [-2 step] → 550 [-4 step] → 250 [-4 step, clamped]) where the previous algorithm needed 5+ pure-loss games. For an agent calibrating against the right opponent (alternating W/L), the streak stays near zero and the algorithm degenerates to single-step behaviour — exactly what we want for the steady-state measurement phase.

Both behaviours were verified with a simulated 20-game sequence using the true Elo win-probability function: an agent of true ELO 300 calibrated to the 250–400 chess.com band by game 8.

### Initial ELO rationale, now documented

The user asked whether 1200 is the standard starting ELO. The honest answer (documented in the decision record's revised §2): **1200 is a convention, not a canonical value.** Different bodies use different anchors:

- USCF historically uses 1200 for unrated scholastic players.
- Chess.com starts new accounts at 800 (rapid) with fast calibration.
- Lichess uses 1500 as the global mean.
- The chess literature uses 1500 for "average tournament player," 1000–1200 for "casual amateur."

We anchor at **1200** because:

- It's at the lower end of "uncalibrated amateur," which doesn't overstate expected skill — important when the agent's true skill is below most playable opponents.
- It sits at exactly one of the discrete grid points in both pools (Maia 1200, chess.com 1200 = slider position 8 of 25), so the first game has no rounding artefacts.
- The starting point only affects calibration speed, not the converged estimate; documenting the choice and the rationale is what matters for the thesis.
