# 2026-05-25 â€” Aborted game investigation: Gemma 4 tool-calling behaviour on illegal-move recovery

## Context

The fourth game of the first ranked calibration batch (opponent: chess.com 550 ELO, batch "elo calibration") was aborted with `aborted_reason = "Agent did not successfully call make_move.py this turn."`. The prior three games (vs 1200, 1100, 1000) completed normally â€” the agent called `make_move.py` successfully every turn. This failure is specific to this game and this turn.

## What the trace shows

The aborted turn is turn 21 of game `bae5639fc5b044dcac842d14c8ae79fd`. FEN at time of failure:

```
1n1qk2r/rp2bp1p/p2pb3/8/4B3/2N5/PPP2PPP/R2QK2R w KQk - 0 11
```

Black has just played `Bxe6` (a bishop capture, landing a black bishop on e6). The agent sees a white bishop on e4 and a black bishop on e6 and wants to capture: `e4e6`.

**Event sequence in the failing turn:**

1. Model responds with `use_skill(chess-player)` + `run_script(list_legal_moves.py)` in one batch.
2. Legal moves come back. The list does **not** include `e4e6`.
3. The model streams ~1000 characters of reasoning. It correctly notices that `e4e6` is absent from the list, re-reads the FEN, re-checks its logic, and decides to fall back to `e1g1` (castling) as the alternative.
4. Model then calls `run_script(make_move.py --uci e4e6)` â€” **still the original illegal move, not the fallback it just reasoned about**.
5. `make_move.py` returns `{"ok": false, "error": "'e4e6' is not a legal move. Legal moves: [...]"}`.
6. Model streams another large block of text re-examining the situation and again reasons its way to `e1g1`.
7. Model calls `run_script(make_move.py --uci e4e6)` **again** â€” the same illegal move a second time.
8. Second failure returned. Model streams more text. At no point does it call `make_move.py` with a legal UCI.
9. `max_turns=20` is set in `AgentConfig`; the runner hits the request limit and the run ends without a successful `make_move.py`. `AgentPlayer.get_move()` raises `PlayerError("Agent did not successfully call make_move.py this turn.")`.

## Root cause analysis

There are three layered issues:

### Issue 1 â€” Gemma 4 does not naturally translate chain-of-thought into a corrected tool call

GPT-4o reliably writes out the reasoning, then calls the corrected tool. Gemma 4 appears to use a **separate internal "thought" channel** (observable as `<|channel>thought<channel|>` markers in the streamed text) rather than in-context chain-of-thought. The conclusion it reaches in the thought channel does not always propagate forward to the tool call that follows. This is not a prompt-adherence failure in the usual sense â€” the model clearly reaches the right answer in text â€” but the reasoning and the tool-dispatch are not tightly coupled the way they are in GPT-4o.

This is a model-specific behaviour difference. GPT-4o's tool-calling was tested and tuned against a system where the text output feeds the next action. Gemma 4's internal reasoning channel is opaque to the prompt machinery.

### Issue 2 â€” `max_turns=20` is a hard cap on model requests, not on game turns

`AgentConfig(max_turns=20)` maps to `UsageLimits(request_limit=20)` in pydantic-ai. This counts each model request (each time the LLM is called, whether it produces a tool call or text), not each chess turn. Turn 21 has 4 tool calls, which implies at least 3â€“4 model requests just for that one chess ply. If a turn uses several requests reasoning and retrying, the cap can be exhausted across a session rather than per-turn.

**Important:** The cap was not reached in this game â€” the failure was that after both `make_move.py` refusals the model ran out of reasoning but never called `make_move.py` with a legal move. The stream ended normally (the `AgentRunResultEvent` fired). The `max_turns` cap is a parallel latent risk but it is not what caused this particular abort.

### Issue 3 â€” No harness-level failsafe

When the run ends without a successful `make_move.py`, `AgentPlayer.get_move()` raises `PlayerError` and the game is recorded as aborted. There is no retry loop at the `AgentPlayer` level: if the model fails to submit a legal move, the turn is simply lost and the game is over. With GPT-4o this was never triggered; with Gemma 4 it is triggered immediately when the model contradicts its own reasoning.

## Why it worked with GPT-4o

GPT-4o's tool use is tightly coupled to its text generation: when it writes "I will play e1g1", the next action is typically `make_move.py --uci e1g1`. The model's text output and its tool call queue are effectively the same stream. Gemma 4 separates these: `<|channel>thought<channel|>` content appears not to flow into the model's action selection the same way in-context text does for GPT-4o.

## What needs to change

Two complementary fixes address the problem:

**Fix A â€” Harness-level retry in `AgentPlayer.get_move()`:**  
After `run_stream` ends without a successful `make_move.py`, do not immediately raise `PlayerError`. Instead, re-invoke `run_stream` with an explicit follow-up prompt: `"You have not submitted a move yet. Legal moves are: {legal_moves}. Choose one and call make_move.py now."` This gives Gemma 4 a clean shot at the task with no accumulated history confusion. Retry once (or at most twice) before giving up.

This is a harness-level failsafe, not a model instruction change. It works regardless of the model's internal reasoning architecture.

**Fix B â€” System prompt adjustment for Gemma 4's tool-calling pattern:**  
The current `system_prompt_extra` ends with `"make_move.py is always the last action of your turn."` For GPT-4o this is sufficient. For Gemma 4, a stronger instruction may help: make explicit that when `make_move.py` returns `{"ok": false}`, the model **must call `make_move.py` again with a different move from the legal-moves list before ending the turn**. Something like: `"If make_move.py returns ok=false, immediately call it again with a different move from the legal moves list."` This reduces reliance on the thought-channel completing correctly.

Fix A is the higher-priority because it is a mechanical guarantee; Fix B is complementary and cheaper to add.

**Not a fix to consider:**  
Reducing `max_turns` further, or making it per-chess-turn, addresses a different risk. The immediate problem is that the agent reasons correctly but fails to act on its reasoning. A tighter turn cap would only add a new abort path.

## Open questions

- Is this thought-channel / tool-dispatch decoupling a Gemma 4 architectural property, or does it vary with temperature or formatting? Worth checking with temperature 0 in an experimental game.
- Does the `<|channel>thought<channel|>` text appear in the context window that subsequent model requests see? If not, the second and third requests are starting from a less informative state than it appears.

## Implementation: what was actually changed

The fix went further than the two-fix proposal above â€” it addresses the structural root cause rather than adding a workaround layer.

### make_move.py now commits the move

`make_move.py` was a pure validator: it fetched legal moves, checked legality, printed `{"ok": true, "move": "e2e4"}`, and exited. The backend applied the move later (after `get_move()` returned). This separation was the fundamental problem: the agent could "succeed" at the tool call without anything actually happening on the board.

`make_move.py` now POSTs to `POST /api/games/{id}/agent-move`, a new backend endpoint. The move is applied to the board immediately when the tool returns. When the tool succeeds, the turn is genuinely over â€” not because the harness decided to stop reading, but because the board state changed.

On failure, `make_move.py` now returns the current legal moves list in the error response so the model can pick a valid move without a separate `list_legal_moves.py` call.

### The harness breaks on committed move, not on stream state

`AgentPlayer.get_move()` now breaks the stream as soon as any `run_script(make_move.py)` result shows `ok=true` in the tool log. The model can call `list_legal_moves.py` any number of times, write as much reasoning as it wants, retry `make_move.py` on failures â€” the harness simply waits for a committed move. No more "last action" fiction in the prompt.

### Retry loop on stream exhaustion

If `run_stream` ends without a committed move (Gemma 4's thought-channel / tool-dispatch decoupling scenario), `get_move()` retries up to `_MAX_MOVE_RETRIES=2` times with a fresh context. Each retry clears conversation history and sends a new prompt: original position + explicit legal moves list. This is the mechanical failsafe â€” it works regardless of the model's internal reasoning architecture.

### Double-apply prevention in the bot loop

`_run_until_human_or_finished` previously called `_push_move(game, move)` unconditionally after `get_move()` returned. Now that `make_move.py` may have already applied the move, the bot loop checks `len(game.uci_moves) >= move_number` before pushing. If the move is already on the board, it skips the push and proceeds.

- `backend/skills/chess-player/scripts/make_move.py` â€” POSTs to `/agent-move`; returns legal moves on failure
- `backend/skills/chess-player/SKILL.md` â€” documents the commit-on-call contract; clarifies retry behaviour
- `backend/app/game_service.py` â€” new `submit_agent_move()` method; double-apply guard in bot loop
