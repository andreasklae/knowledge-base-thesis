# 2026-05-27 — Post-merge calibration batch: observed failure modes + harness question

The `visualization-and-context-management` branch is merged to main
([[2026-05-26-branch-wrapup]]). Running batches now on main to
calibrate ELO on the new baseline. The agent is qualitatively much
stronger than the pre-branch baseline (perception tools + step-0 mate
scan + king mobility — see [[2026-05-26-endgame-fixes]]) but still
makes systematic mistakes in patterns that look fixable. Five
observations from watching this batch, plus a structural question
about whether skillful-agent is still the right substrate for Gemma 4.

## 1. Hanging mate-in-one (and pieces left hanging despite the warning)

The agent has *played a move that allows mate next turn* in multiple
games this batch. Mate-in-1 by the opponent should be the lowest-
hanging fruit possible to add — after every candidate move the agent
imagines, run a one-ply scan: does any legal reply produce `is_checkmate()`?
If yes, flag it in the `imagine_move` output with the same prominence
as the existing king-mobility line.

The broader pattern: the agent visualizes its own move well but
under-visualizes the *opponent's* reply. The replies-table change on
the stabilization branch ([[2026-05-26-stabilization]] — full table
instead of top-12) helped, but the agent still doesn't seem to use it
as a checking step. May need a stronger framing: explicitly enumerate
"opponent's best replies" with each tagged for tactical motif (mate,
fork, capture of hanging piece) rather than relying on the agent to
read the table and notice.

**Pieces left hanging despite warning.** Already a known limitation
([[2026-05-26-branch-wrapup]] session 3 logged the value-blind
hanging detection for `count(attackers) == count(defenders)` cases).
This batch confirms it's not just edge-case behaviour — the agent
will play a hanging move even when the existing warning fires. So
two separate problems stacked: (a) the warning has false
*negatives* on equal-count exchanges where the defender is more
valuable than the attacker, and (b) the warning, when it *does*
fire, sometimes isn't acted on. (a) is the SEE-lite follow-up
flagged in [[2026-05-26-branch-wrapup]] session 3. (b) is harder —
maybe a stop-the-line affordance: if hanging is detected and severity
is high, surface it *before* the candidate evaluation and refuse to
let the agent commit without an acknowledgement.

**False positives on protected pieces.** New observation. The
hanging-piece warning is firing on pieces that are in fact protected.
Need to actually read the logs and confirm before guessing a cause —
could be the attack-chain computation missing a defender (battery
behind a friendly piece?), could be `is_attacked_by` semantics around
pinned defenders, could be a recent change to `_moved_piece_hanging_warning`
that overcorrected. Add to follow-up: pull 3-4 instances from this
batch's logs and trace through.

## 2. `imagine_move` failure loop — missing argument

This is the most concerning of the five. The agent calls
`imagine_move` without a required argument, gets a clear error
message back ("missing argument: move" or similar), and **does not
self-correct**. It re-issues the same incomplete call, gets the same
feedback, and loops. The other tools (`list_legal_moves`,
`show_position`, `make_move`) don't show this pattern.

Hypothesis space:

- **Prompt / SKILL.md asymmetry.** `imagine_move` may be described
  differently from the other tools in SKILL.md or in its argument
  schema description, in a way that confuses the model about whether
  the move arg is required or what format it takes.
- **Tool-script signature.** The script's argparse / pydantic-ai
  schema may be inconsistent with the others (e.g. positional vs
  flag, different name conventions for the move parameter).
- **Stale error text.** If the error string doesn't actually name
  the missing argument clearly, Gemma 4 (E4B class) may not be able
  to repair from it.

Action: read the actual logs for one of these loops, compare the
*exact* surface (`SKILL.md` text + script signature + error message)
of `imagine_move` against `show_position` and `make_move` side-by-
side, normalize. The brief should be: every tool follows the same
argument convention, the same SKILL.md framing, the same error
shape. Convention drift across four scripts is the most likely
culprit and the cheapest fix.

## 3. Misses forks

The agent doesn't see forks — neither when it could play one nor when
the opponent threatens one. The current perception toolkit has check,
checkmate, hanging, and king-mobility detection; no fork detection.

Adding it isn't trivially cheap (a fork is "one piece attacks two or
more enemy pieces of which at least one is undefended or higher-
valued") but it's mechanical: for each piece on the board, list the
squares it attacks; for each attacking piece, count how many enemy
pieces it forks. Expose in `imagine_move`'s output as a `Forks:`
line, and ideally also as a column in `list_legal_moves` (mirroring
the king-mobility column).

Worth thinking about whether to surface *threatened* forks from the
opponent's perspective the same way — same calculation, run on the
post-move position with side-to-move flipped.

## 4. Still ineffective at finding mates

Even with step-0 mate scan ([[2026-05-26-endgame-fixes]]), the agent
misses non-immediate mates. The user's framing is exactly right:
the agent needs hints like *"Qe5 would be mate if e5 was protected"*
— i.e. tell it which moves are *one defender away* or *one
support-piece-move away* from being mate.

This is a more interesting capability than the others because it's
the first one that looks *concept-shaped* rather than
percept-shaped. Detecting "this would be mate if X" requires:

- Generating candidate "near-mate" squares (any square the enemy
  king must escape to, or any check that limits king mobility to 0
  with one undefended exit).
- For each, identifying the missing condition (this square is
  attacked by the enemy, this checking piece is undefended).
- Reporting in language the agent can act on: *"Bxd4 sets up Qd7#
  if the queen reaches d7 with the e-file clear."*

This is also where the deterministic-tools-hypothesis gets
interesting [[deterministic-tools-hypothesis]]. We're not asking the
agent to *find* the mate, we're asking the perception layer to
surface the candidate. Whether that crosses the line into "the tool
is playing the game" is a real question for the thesis discussion;
arguably the chess-engine PV would be the cleaner deterministic
delegation, and we're inching toward it with each perception addition.

## 5. Skill loaded every turn

Observed: `use_skill` is called once per turn, even though SKILL.md
content presumably doesn't change. Not an issue *per se* (single
tool call per turn, ~one chunk of context), but unnecessary.

This is actually a context-management hint. The per-turn fresh-
context model ([[2026-05-24-per-turn-fresh-context]]) means SKILL.md
is reloaded on every turn by design. We could either:

- Pre-inject SKILL.md into the system prompt at agent construction
  and remove `use_skill` from the tool list for this agent (matches
  the disabled-tools pattern already used for `manage_todos` etc.).
- Keep `use_skill` available but make its first call cached / no-op
  for the agent's lifetime.

Option 1 is cleaner. Worth a quick check that the agent doesn't
*rely* on the use_skill round-trip as a thinking beat — sometimes a
mandatory step early in the turn is doing more than its surface
function suggests.

## The harness question

User pasted two external references and an explicit "I'm open to
using something else than skillful-agent (maybe make a new one?)
that is better suited for Gemma":

- [[mirko2026gemma]] — Mirko's recipe
  generator. Key takeaway: Pydantic AI's default `ToolOutput`
  pattern is unreliable with Gemma 4 via Ollama; switching to
  `NativeOutput` (Ollama's `format=` JSON schema, server-side
  constrained decoding) plus temperature 0.2 makes it reliable.
  E4B (4.5B) reported as the most reliable Gemma variant for
  programmatic use.
- [[mayo2026gemma4tools]] — Mayo's KDnuggets
  tutorial. From-scratch tool-calling loop with `gemma4:e2b` via
  Ollama. Two-pass synthesis: query → dispatch tools → query again
  with tool results as `role: tool` messages. The companion Google
  notebook documents Gemma's actual chat template
  (`<|tool_call>call:func{arg:<|"|>val<|"|>}<tool_call|>`) and the
  `enable_thinking=True` parsing pattern.
- The `pydantic/pydantic-ai-harness` GitHub repo as a third
  reference point.

The question I want to think through here, not answer:

**Is skillful-agent fighting Gemma 4?** The 400 BadRequest from vLLM
on malformed tool-call JSON ([[2026-05-26-stabilization]] problem 2)
is exactly the failure mode `NativeOutput`-style constrained
decoding eliminates. The `<|channel>thought<channel|>` framing
breakage ([[2026-05-26-stabilization]] problem 3) was specific to
Gemma's chain-of-thought channel and *also* specific to how
skillful-agent / pydantic-ai surfaces streamed reasoning. The
`use_skill` reload every turn is structural to skillful-agent's
model. And the `imagine_move` argument-loop above may be a
tool-call-format-fragility issue that constrained decoding would
fix.

Against switching: skillful-agent has paid for itself in this
experiment. The per-turn fresh-context model, the `disabled_tools`
mechanism, the SSE replay, the tool/skill registry — these are
infrastructure I've already built and validated. The thesis
*itself* is about productivity as an infrastructure problem
[[agent-infrastructure-vs-capability]]; replacing the harness mid-
experiment is the kind of move that re-opens questions the
experiment design has already closed. And N=1 article from a recipe
generator is not strong evidence the structural problem is the
harness rather than the prompts.

A more modest version of the change that might capture the win
without the cost: keep skillful-agent, but switch the underlying
LLM call to use Ollama's `format=` (or vLLM's `guided_json`) for
constrained tool-argument decoding. The stabilization diary already
flagged `guided_json` as the natural next move on stability
([[2026-05-26-stabilization]] "Reflection on what stayed out of
scope"). The Mirko post is independent evidence pointing the same
way.

**Decision deferred.** Not making this call in a diary entry. The
right next step is:

1. Get a clean log of the `imagine_move` argument-loop and confirm
   whether it's a constrained-decoding-shaped problem or a SKILL.md
   asymmetry.
2. If constrained-decoding-shaped: write up an ADR proposing
   `guided_json` (vLLM) for tool arguments, keeping skillful-agent
   for the rest. Reference both this diary and
   [[2026-05-26-stabilization]] as the failure-mode evidence.
3. If SKILL.md-asymmetry-shaped: fix the conventions and move on;
   the harness question reduces to "interesting reference, not now."
