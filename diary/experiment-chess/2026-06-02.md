# 2026-06-02 — Typed-tool harness verified; agent knowledge-wiki designed and scaffolded

## What happened

Completed the typed-tool harness implementation in `skillful-agent` and ran a full chess game against it. Everything worked as intended.

The key implementation pieces landed yesterday and today:

- Static AST extraction of argparse schemas into `ToolSpec` objects at registry parse time
- Per-script typed tools registered upfront via `Tool.from_schema`, gated by a `prepare` closure that checks `activated_skills`
- `activated_skills` lifecycle fixed — persists across `run()` calls, resets only on `clear_conversation()`
- `run_script` removed entirely — the untyped dispatcher is gone
- `cwd=None` bug in subprocess execution fixed; `PYTHONPATH` now includes both skill root and `scripts/`

## The prepare-gating mechanism in practice

One design question resolved during implementation: whether `use_skill` needed to raise `ModelRetry` to force a fresh model request before the skill's typed tools would be visible. Initial implementation did raise `ModelRetry`; after review this turned out to be redundant.

pydantic-ai re-evaluates `prepare` on every step (every `CallToolsNode → ModelRequestNode` transition), not once per run. When `use_skill` returns normally, the tool-call loop produces the next request automatically, and by that point `activated_skills` already includes the newly loaded skill. The `ModelRetry` was removed. The probe log confirms it:

- Turn 2, Request 1: `greeter__greet` absent (skill not yet loaded)
- Turn 2, Request 2: `greeter__greet` present — normal `use_skill` return, `prepare` re-ran, gate opened

No retry, no fake error in history, no retry-budget consumption.

## Chess game result

Ran a full game with the new harness on eX3 (Gemma 4 31B, hgx2q, TP=2). The agent called `use_skill("chess")` and then used the typed chess tools directly — typed args, no `args[]` list. The Gemma 4 tool-call parser handled the mid-conversation tool-set change without issue. The eX3 note in the decision record about needing to verify this can now be considered resolved.

## What this means for the experiment

`run_script` was the last source of untyped tool calls in the chess agent. The model now operates entirely in native tool-calling format — the format it is fine-tuned for. This should reduce argument assembly errors and removes one category of failure mode from the experiment data.

The `activated_skills` persistence change is the one behavioural break worth watching: skills loaded in turn N are now visible in turn N+1. For chess this is fine (the chess skill is loaded once and should stay loaded), but it is a semantic change in the harness that any future experiment should be aware of.

## Later that day — designing and scaffolding the agent's knowledge wiki

Separate strand of work: turned the long-planned "agent's own LLM wiki" (sketched in [[experiment-chess]] and [[skill-acquisition-loop]]) from a plan into an actual scaffold, after a long design discussion.

### Decisions made (now ADRs)

- **[[2026-06-02-chess-agent-wiki-architecture]]** — the wiki *is* the chess skill's `references/` directory. Folders: `openings/`, `principles/` (short rules of thumb), `strategic-thinking/` (long-term planning, with `pawn-structures/` beneath), `patterns/` (tactical, with `mating-patterns/`), `endgames/`, `game-analyses/`. Each folder has an `index.md` that is a *routing decision-tree*, not a table of contents. Pages capped ~400 words with a fixed shape (When to use / The idea / What to do / Watch out for / Examples) and frontmatter (`description`, `triggers`, `tags`, `related_pages`, `status: draft→tested`). Read-only during play; the tutor maintains it at batch boundaries.
- **[[2026-06-02-tool-fairness-rulebook]]** — promoted the tool-fairness rule from the work page / [[tool-fairness]] concept page into a citable ADR. Mechanics tools always allowed; retrieval allowed iff the corpus was built through the loop; engines/Lichess post-game only. Crucially this **licenses pre-seeding the wiki with synthesised theory** under the reading-and-noting clause.

The big framing shift in the discussion: the experiment is **tutored deliberate practice**, not zero-start self-play. I (the tutor) point the agent at material, give it tools, say what to read; it accumulates a compounding artefact. The KB's earlier "infinite analyse-and-improve loop" framing was misleading and is now corrected on the work page.

### What I built

- Wiki skeleton under `experiments/chess/backend/skills/chess/references/`: top routing `index.md`, `log.md`, six folder indexes (each with a "Read with" column showing the exact `read_reference` call), and one verified seed page — `patterns/mating-patterns/back-rank-mate.md` — whose example FEN I checked is genuinely mate in python-chess. This validated the page contract end-to-end.
- `scripts/search_wiki.py` (→ typed tool `chess__search_wiki`): keyword search returning each hit's path + frontmatter + the exact `read_reference` call to open it, never the body.
- Rewrote the chess `SKILL.md` "Your knowledge wiki" section + turn-workflow step; rewrote the chess `CLAUDE.md` with a wiki-maintenance section casting the coding agent / me as the tutor-maintainer (ingest / post-game / lint operations, the page contract, a verify-every-claim rule).

### Harness reality check that reshaped the design

My initial scaffold assumed `read_reference` could be gated by listing index files in SKILL.md frontmatter, and that the agent would call scripts via `run_script(...)`. Both were wrong against the current harness:

- The harness's `read_reference` took a flat top-level *filename*, ignored the `references:` frontmatter, and couldn't reach subfolders — so a nested wiki was unreachable. **Fixed in `skillful-agent`** (`435fa8d`): `read_reference` now takes a `path`, supports subfolders, and is path-jailed instead of allowlisted. Dropped my transient `read_wiki.py`; `read_reference` is the page reader.
- `run_script` was already gone (typed tools). Rewrote SKILL.md to the `chess__make_move(move=, reasoning=)` style, fixed the backend's move-detection (`_committed_move_from_result` now keys on `chess__make_move`, not a `run_script` + filename check — this had been silently broken by the harness change), and fixed runtime error strings in `make_move.py`/`imagine_move.py` that told the agent to retry with dead syntax.

### Cleanups in the same pass

- Slimmed the SDK system prompt to harness-essentials and removed its stale `run_script` description; made the disabled-tool stripper renumber rules and collapse blanks (`188c841`, `556af8e`); added `test_read_reference.py` and `test_system_prompt.py`.
- Contradiction sweep for decisions we've moved away from. Beyond the `run_script` references (SDK CLAUDE.md, README, both native skills), found a real in-code one: `elo.py`'s module docstring still described the **doubling** matchmaking rule that `step_from_streak` in the same file says it *replaced* with single-step ([[2026-05-26-single-step-matchmaking]]). Fixed that docstring and a matching pointer in `batch_service.py`.

### Where this leaves the experiment

The wiki infrastructure is in place and validated, but empty apart from one seed page. Next session: ingest real content into `strategic-thinking/` and `patterns/mating-patterns/` — the two categories we agreed to start with — either from source material or synthesised from general theory.
