# 2026-05-19 — Agent integration, logging, skill rethink, and the chesscom-driver package

## What was built today

Three major additions to the chess experiment, plus an architectural correction that came out of a review mid-session.

### 1. AgentPlayer and skillful-agent integration

Added a third `Player` implementation, `AgentPlayer`, backed by the `skillful-agent` SDK imported directly as a Python library dependency (no separate server process). The SDK is declared as a Git dependency in `pyproject.toml` pointing at the public GitHub repo; `uv sync` resolves it cleanly alongside the existing FastAPI/python-chess stack.

The agent is wired with Azure OpenAI (gpt-4o-mini) via the credentials in the skillful-agent `.env`. The `AgentPlayer` constructs its own `pydantic-ai` model + provider and passes them into `Agent(model=model, skills_dir=...)`. This is the intended import pattern: `skillful-agent` is a library, not a service; the caller owns the model and just hands it to `Agent`. The server entrypoint in the skillful-agent repo is one consumer of this pattern, not the canonical one.

The agent currently plays white only, consistent with the experiment design described in [[experiment-chess]].

### 2. Chess tools: first attempt and architectural rethink

The initial implementation injected `list_legal_moves` and `make_move` as pydantic-ai tools directly into the skillful-agent runner via `@agent._runner.tool_plain`. This works mechanically — pydantic-ai supports post-init tool registration — but it bypasses the skill system entirely. The tools were invisible to the agent until it called them; the skill body gave instructions but the implementation lived outside the skill.

After review, the better design is to put `list_legal_moves.py` and `make_move.py` as scripts inside the `chess-player` skill directory, accessible via the `run_script` mechanism already built into skillful-agent. The agent loads the skill (which discloses the scripts), then calls `run_script("chess-player", "list_legal_moves.py", "--game-id <id>")` etc. The scripts call the chess backend's HTTP API to get game state and submit moves.

This is the more principled approach because:
- The skill is fully self-contained — instructions and tools travel together.
- It respects the progressive disclosure architecture: the agent sees the script names only after loading the skill, exactly as designed.
- It's a direct test of the [[skills-component]] claim that skills package procedural knowledge alongside the agent's capability to execute it.
- The backend API already exposes everything the scripts need (`GET /api/games/{id}`, `POST /api/games/{id}/moves`); the scripts are just thin clients.

The injected tools in `agent_player.py` are now scheduled for removal in favour of this approach. `API_BASE` and `game_id` will be passed as CLI arguments to the scripts.

### 3. Logging: games.csv and per-game agent JSON

Implemented `LoggingService` in `app/logging_service.py`. Two artefact types:

**`games/games.csv`** — append-only, one row per game. Columns match the logging schema from [[experiment-chess]]: `game_id`, `batch_id`, `datetime`, `white_type`, `black_type`, `opponent`, `result`, `elo_before`, `elo_after`, `skill_repo_sha`, `model`, `temperature`. Added one extra column: `agent_log_path` — the relative path to the per-game JSON trace for agent games; empty string for non-agent games. The CSV is the primary query surface for Elo tracking and batch analysis. `pandas` can load it directly.

**`games/<game_id>_agent.json`** — one per agent game. Structured by turn: each turn records the prompt given to the agent, the full event stream (text deltas, tool calls, tool results), and the move chosen. Written at game end via `LoggingService.finalize`. This is the primary artefact for qualitative analysis of agent reasoning — what the agent said, what tools it called, and whether the move matched its stated analysis.

The `agent_log_path` column in the CSV is what links the two. A row with `agent_log_path=""` was a non-agent game; a row with a value points to the reasoning trace.

### 4. Agent SSE endpoint

Added `GET /api/games/{game_id}/agent-events` as a second SSE channel alongside the existing board-state channel (`GET /api/games/{game_id}/events`). The board channel emits `event: state` payloads (full `GameState` JSON). The agent channel emits `event: agent` payloads — individual streaming events: text deltas, tool calls, tool results, and the incoming prompt. These are separate channels because they serve different consumers at different rates: the board channel updates the chessboard widget; the agent channel feeds the chat panel.

The `Game` dataclass gained an `agent_event_subscribers` set alongside `subscribers`, following the same fan-out pattern.

### 5. Frontend: AgentPanel component

Added `AgentPanel.tsx` — a scrollable chat feed rendered to the right of the board when the white player is `agent`. The panel connects to the agent SSE channel and renders four event types with distinct visual treatment:

| Event | Visual |
|---|---|
| `prompt` | Green-tinted bubble labelled "prompt" — shows the FEN/context the agent received |
| `text_delta` | Cream bubble labelled "thinking" — streams in real time with a blinking cursor while incomplete |
| `tool_call` | Blue-tinted entry labelled "tool" — shows `🔧 list_legal_moves` or `🔧 make_move(uci: e2e4)` |
| `tool_result` | Green-tinted entry labelled "result" — `✓ 28 legal moves` or `✓ played e2e4` |

The `game-layout` grows from 2 to 3 columns when white is `agent` (board | status | agent). On screens narrower than 1100px it collapses back to 2 columns with the agent panel spanning the full width below.

### 6. Player type: 'agent' added throughout

`schemas.py`, `players.py`, `PlayerFactory`, frontend `types.ts`, `LobbyPage.tsx`, and `api.ts` all updated to treat `agent` as a first-class player type. The lobby's player selector now shows `agent` as an option for white; no Elo required.

## Reflection on the architecture question

The most interesting design moment today was the tool injection question. The initial approach was pragmatic — inject tools into the pydantic-ai runner, it works. But the more principled approach is to put the tools in the skill's `scripts/` directory and have the agent call `run_script`. This aligns with the experiment's own thesis claim: that procedural capability packaged into skills is the unit of agent infrastructure. If `list_legal_moves` and `make_move` live outside the skill, the skill is just documentation, not capability. If they live inside the skill as scripts, the skill genuinely encapsulates what the agent can do.

This is a small example of the [[deterministic-tools-hypothesis]] in action: the chess API is a fully deterministic tool (it either accepts or rejects a move, with no ambiguity), and routing it through a skill script rather than a bespoke tool injection is the methodologically clean version. The experiment will ultimately test whether having the skill at all (Config 2) beats no skill (Config 1); having the scripts live in the skill is what makes that comparison clean.

The injected-tool approach also has a subtle problem for reproducibility: `skill_repo_sha` in the CSV is supposed to identify the exact agent configuration that played each game. If the tools live in the skill's scripts directory, the SHA captures them; if they live in `agent_player.py`, they don't.

## Move detection fix: ToolResultEvent has no payload

Later in the session, after moving the tools into skill scripts, a subtle SDK issue surfaced. `ToolResultEvent` — the event yielded by `skillful-agent`'s `run_stream()` when a tool call completes — only carries `name: str`. There is no content, stdout, or result payload on the event object. The original `_extract_move_from_events` function was looking for `event.get("result")` on these events and would always return `None`, meaning `get_move()` would always raise `PlayerError`.

The fix is to read from `agent._deps.tool_log` instead. The SDK maintains a `list[ToolCallRecord]` in `_deps` throughout the run; each `ToolCallRecord` has `tool`, `input` (the args the LLM passed), and `output_preview` (first 500 chars of the return value). For `run_script` calls, the return value is the JSON wrapper `{"ok": bool, "stdout": "...", "stderr": "...", "exit_code": N}` that `run_script` always produces. `make_move.py` prints `{"ok": true, "move": "e2e4"}` to stdout, so the move is recoverable by parsing two layers of JSON from `output_preview`.

`_extract_move_from_tool_log` replaces `_extract_move_from_events` and is called after the stream completes (at which point `tool_log` is fully populated). The `ToolResultEvent` handler in the streaming loop now also consults `tool_log` in real time to attach a result preview to SSE events for the `AgentPanel` — so the frontend can show "✓ played e2e4" rather than a blank result.

This is also a useful observation about `skillful-agent`'s streaming API: `ToolResultEvent` is a signal that a tool finished, not a carrier of its output. If you need the output in streaming context, you read it from `_deps.tool_log` (the most recent record matching the tool name).

## Further improvements: ergonomics and UX polish

Several smaller improvements made after the first game attempt:

### Script context injection

The original script interface required the agent to pass `--api-base` and `--game-id` as CLI arguments, which meant the prompt had to include those values and the SKILL.md had to explain how to thread them through. This was awkward — the agent was being asked to relay bookkeeping that the backend already knows.

The fix: `_build_agent()` now sets `CHESS_GAME_ID` and `CHESS_API_BASE` in the process environment before constructing the agent. The scripts read these from `os.environ` instead. The call site in SKILL.md simplifies to `run_script("chess-player", "list_legal_moves.py", "")` and `run_script("chess-player", "make_move.py", "--uci <move>")`. The agent only needs to supply the one thing it actually decides: the move.

### Prompt now includes the actual opponent move

`get_move()` now accepts a `last_san` parameter passed from `game_service._run_until_human_or_finished`. The prompt is "Opponent played Nf3. Current position (FEN): ..." rather than a generic "New move." This gives the agent concrete information about what just happened — useful for evaluating threats and for qualitative analysis of whether the agent's reasoning responds to the actual game continuation.

The `Player` ABC's `get_move` signature was updated to accept `last_san: str | None = None`, keeping all existing implementations (HumanPlayer, MaiaPlayer) compatible without change.

### Agent log: coalesced thinking

The raw agent log was recording individual `text_delta` events — one per token — which made the JSON hard to read. Text deltas are now accumulated in a buffer during streaming and flushed as a single `{"type": "thinking", "content": "..."}` event each time a tool call or the run end interrupts the text stream. The SSE stream to the frontend still sends individual deltas for the live streaming effect; the coalesced version only appears in the persisted JSON log.

### Promotion picker

The frontend was auto-selecting queen on pawn promotions. A modal overlay now appears when the human drags a pawn to the back rank, offering Queen / Rook / Bishop / Knight. Detection uses `isPromotion()` which checks whether any 5-character legal move starts with the from→to prefix. For the agent, no change was needed — python-chess already emits all four promotion variants in `legal_moves` (`e7e8q`, `e7e8r`, `e7e8b`, `e7e8n`).

### AgentPanel UI improvements

`AgentPanel.tsx` now parses the raw `run_script` tool call args to display friendly strings: `list_legal_moves()` and `make_move(e2e4)` instead of the raw skillful-agent invocation with JSON args. Tool results are similarly parsed — the two-layer JSON (`run_script` wrapper → script stdout) is unwound to show "28 legal moves" or "played e2e4" or an error message directly.

### .env location fix

Azure credentials were being loaded from `software/skillful-agent/.env` via a relative path that resolved incorrectly (4 levels up instead of 3, missing the `v2/` directory). The credentials are now in `experiments/chess/backend/.env` and the path constant in `config.py` points directly there.

## Later in the day: the chesscom-driver package

The opponent-pool gap that's been hanging over the experiment finally got addressed. Maia covers 1100-1900 ELO. The agent (gpt-4o-mini in its current configuration) plays around 300 ELO raw. There is no opponent in the existing pool for the bottom of the agent's range. Lichess has bots down to ~250 but every game on Lichess requires a time control, and time controls are excluded from the experiment (see [[experiment-chess]] — agent move latency is dominated by API/network, not chess-skill time management, so timing the agent measures the wrong thing).

Chess.com has Engine bots covering 250-3200 ELO with no time control. No public API, so the only path is a browser. Cloudflare blocks naked HTTP scraping aggressively, but a real Chrome with a persistent login cookie passes through with no interference.

### Discovery phase

Started by reading two existing chess.com bot repos for reference, then probed the live page via the Chrome MCP. Found that the page exposes a substantial JS API on `wc-chess-board.board.game`:

- `getFEN()`, `getLastMove()`, `getHistorySANs()`, `getTurn()`, `getPlayingAs()`, `isGameOver()`, `getCalculatedResult()`.
- `setOptions({autoPromote: true})` — auto-queening.
- `move({from, to})` — updates internal state.
- `onAll(cb)` — event subscription.

Two empirical findings that shaped the design:

1. **`board.game.move()` does not trigger the bot's reply.** It updates the board state internally but chess.com's bot only responds to genuine pointer/mouse events on board pixels. The driver therefore submits moves by dispatching `pointerdown`+`mousedown`+`pointerup`+`mouseup` at square pixel centres, not by calling `move()`.

2. **The Engine bot slider is a controlled React component.** Setting `.value` directly is ignored. Have to grab the HTMLInputElement prototype's native setter, call it, then dispatch `input` and `change` events. The React state needs ~400-450 ms to settle before reading back the new rating.

### The 25-position ELO mapping

The Engine bot group has a 25-step slider, `min=1, max=25, step=1`. The position → rating mapping is *not* uniform. Stepped through all 25 positions manually with the slider-settle delay in place and read `.selected-bot-introduction-rating` each time:

```
1=250    6=1000   11=1500  16=2000  21=2500
2=400    7=1100   12=1600  17=2100  22=2600
3=550    8=1200   13=1700  18=2200  23=2800
4=700    9=1300   14=1800  19=2300  24=3000
5=850   10=1400   15=1900  20=2400  25=3200
```

Step sizes: 150 ELO in the beginner band (1-5), 100 ELO in mid-range (6-20), then 100/200/200/200/200 in the master+ band. The table is hard-coded in `chesscom_driver/mapping.py`; chess.com doesn't expose the mapping anywhere queryable and the slider's React debouncing is too aggressive to discover it cheaply at runtime. `closest_position(target_elo)` rounds to the nearest available rating, with ties going to the lower rating (more forgiving opponent).

### Package architecture

Built as a standalone package at `experiments/chess/chesscom-driver/`, sibling to `backend/` and `frontend/`. Two layers:

- **`ChessComDriver`** — low-level browser control. Owns Playwright, the persistent Chrome context, and the page. Methods: `launch()`, `start_game(target_elo) -> GameStart`, `submit_move(chess.Move)`, `wait_for_bot_move() -> chess.Move`, `is_game_over()`, `get_result()`, `close()`. Async context manager.

- **`ChessComPlayer`** — thin adapter matching the experiment's `Player` ABC signature `async def get_move(board, last_san) -> chess.Move`. Holds a driver and an internal `chess.Board` (because the host passes a `stack=False` copy with no history). The first `get_move` call lazily launches the browser and starts a game; subsequent calls submit the agent's move and wait for the bot's reply.

Lazy launch is the right design here. `PlayerFactory.create` is synchronous, but `await driver.launch()` needs an event loop. Deferring the launch to the first `get_move` call avoids having to make the factory async or introduce a separate `setup()` step.

The package has zero runtime dependency on the chess experiment — it imports `chess` and `playwright` only. The backend depends on the package, not the other way around. The `chesscom_driver` import in `app/players.py` is guarded by `try/except ImportError` so the backend still loads if the package isn't installed.

### Persistent profile for login

The driver uses `launch_persistent_context` with a user-supplied directory. First run requires interactive login on chess.com; subsequent runs reuse the cookie. The path is configurable via `CHESS_CHESSCOM_USER_DATA_DIR` (defaults to `backend/chesscom-profile/`). The user-data directory cannot be shared with another running Chrome instance.

### Backend integration

Three additive changes:

- `schemas.py` — added `"chesscom"` to `PlayerConfig.type` and `PlayerTypeInfo.type`, with ELO validation 250-3200.
- `config.py` — added `chesscom_user_data_dir`, `chesscom_headless`, `chesscom_chrome_channel` settings (env prefix `CHESS_`).
- `players.py` — added `ChessComPlayer(_DriverPlayer, Player)` (subclassed for `isinstance` checks) and a `"chesscom"` branch in `PlayerFactory.create`. Validation in `validate_config` raises a clear error if the driver package isn't installed.

### Methodological fit

This sits cleanly inside the [[tool-fairness]] rule. The chess.com bot is an *opponent*, not a tool the agent calls. The driver is server-side infrastructure for running games — equivalent to wrapping Lc0+Maia or Stockfish as a `Player`. The agent has no knowledge that its opponent runs through a browser; it sees only `Opponent played e4. Current position (FEN): ...` in its prompt.

The white-only methodology in [[experiment-chess]] is what made the driver tractable. Supporting black would have required reading the page's side selector, defensively clicking "Play as white", handling the `flipped` CSS class everywhere coordinates are computed, and handling the side-selector remembering the previous game. All of those code paths exist in the JS module but the Python side never exercises them.

### What's verified vs assumed

Verified by interacting with the live page:

- The full `board.game` JS API and the slider control mechanism.
- The full 25-position rating mapping.
- End-to-end flow: set slider to position 2 → "Beginner 400" → click Play → submit e2e4 via clicks → bot responds c5 within ~1.2 s.
- `autoPromote: true` works; `isGameOver()` and `getCalculatedResult()` return sensible values.

Assumed and not yet stress-tested:

- Time control defaults to "No Timer" and stays there across sessions.
- The Engine accordion is open or opens cleanly on first click.
- Side selector defaults to white when starting from the Engine group.
- Resignation flow / new-game flow after a finished game.

These are flagged as known gaps in the README and represent the first round of integration-test work once the package is actually wired into a real game loop.

### Why a separate package and not just a class

The chess.com integration is non-trivial enough to deserve its own package: the JS injection module, the 25-position mapping, the typed exceptions, and the Playwright lifecycle are all conceptually independent of FastAPI and the experiment's player abstraction. Packaging it separately:

- Makes the driver reusable outside this experiment if anyone wants to test agents against chess.com.
- Lets the experiment's developer install or skip it independently (Playwright is a heavy dep — 400+ MB browser binary).
- Keeps the page-DOM contract — the part most likely to break when chess.com redesigns — isolated to one place.

The README documents the four DOM hooks the driver depends on and includes a quick script for re-deriving the slider mapping if chess.com changes step counts.
