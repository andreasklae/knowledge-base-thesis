# 2026-05-24 — serve.py: fully automated launcher, end-to-end verified

Wrote and debugged `software/ex3/serve.py` — a single-file Python CLI that
handles the full eX3 LLM serving lifecycle from the laptop: submit → wait →
tunnel → ready → teardown. Gemma 4 31B-it verified working through the script.

## What we built

`serve.py` is ~270 lines of stdlib-only Python. No packages, no install step.
It presents a small terminal UI (numbered menus), submits an sbatch job, polls
until the node is allocated, waits for vLLM to finish loading, opens the SSH
tunnel, and prints the API endpoint. Ctrl+C cancels the Slurm job and tears
down the tunnel cleanly via a registered SIGINT handler.

Key design choices baked in:

- **Login shell for all SSH calls** (`bash --login` via stdin). Non-interactive
  SSH sessions don't source `/etc/profile.d/*`, so `module` is not on the PATH.
  Piping commands into `bash --login` gives a login shell without interactive
  prompts, making `module load slurm/21.08.8` work from the script.
- **Partition fallback with user prompt.** If the job stays PENDING for >60
  seconds, the script interrupts the polling loop and offers to resubmit on the
  next partition (a100q → hgx2q → dgx2q). The stalled job is cancelled first.
- **Health-check tunnel before the real tunnel.** vLLM takes ~3 minutes to load
  on a warm cache. The script opens a short-lived tunnel on `LOCAL_PORT+1`
  (11501), polls `/v1/models` every 8 seconds, then tears it down and opens the
  real tunnel on 11500. Avoids the port collision between health check and
  production tunnel.
- **HF token from `~/.hf_token`** on eX3, not hardcoded or passed via env from
  the laptop. The sbatch script reads it with `export HF_TOKEN=$(cat ~/.hf_token
  2>/dev/null || true)`. Token never appears in the script or in shell history
  on the laptop. (Previous token was leaked in chat on 2026-05-23 and has been
  revoked.)
- **`%u` instead of `$USER` in SBATCH output paths.** Slurm parses `#SBATCH`
  directives before bash runs, so shell variables are not expanded there.
  `$USER` resolved to the literal string `$USER`, causing the job to fail
  immediately with no log output. Switching to Slurm's `%u` format specifier
  fixed this.

## Bugs hit during development and fixes

### 1. `sbatch: command not found`
Non-interactive SSH sessions don't source the profile. The first version passed
the command as an SSH argument (`ssh ex3 "module load ... && sbatch ..."`), which
works in interactive shells but not here. Fixed by switching `ssh()` to pipe
commands into `bash --login` via stdin.

### 2. Job immediately FAILED, no log output
Slurm `sacct -j <id>` showed exit code 1:0 but the logs directory was empty.
Root cause: `#SBATCH --output=/global/D1/homes/$USER/...` — `$USER` is a shell
variable, not a Slurm format specifier. Slurm tried to open the literal path
`/global/D1/homes/$USER/llm-server/logs/...`, which doesn't exist, and failed
before bash ran. Fixed by replacing `$USER` with `%u` (Slurm's username
specifier) in both `--output` and `--error` directives.

### 3. SSH key passphrase prompt on every call
The script makes several SSH connections (submit, squeue polls, tunnel). Each
was prompting for the key passphrase. Fixed in two steps:
- Added `AddKeysToAgent yes` + `UseKeychain yes` to `~/.ssh/config` under the
  `ex3` host — macOS stores the passphrase in the system keychain after first
  unlock.
- Added `ControlMaster auto` + `ControlPersist 10m` — all SSH calls in a
  session share one authenticated connection.
- One-time: `ssh-add --apple-use-keychain ~/.ssh/id_ed25519` to store the
  passphrase in the keychain.

# 2026-05-24 (continued) — Tool calling, context limits, gh200q, skillful-agent integration

Second session on the same day. Took the working 8k server and extended it with
tool-call support, wired it to skillful-agent, and investigated context limits
across partitions.

## What was done

### 1. Wired skillful-agent to eX3
Added `SKILL_AGENT_EX3_BASE_URL` to `server/config.py` and `server/app.py`. When
set, it takes priority over Azure/OpenAI and uses `OpenAIProvider(base_url=...,
api_key="dummy")`. `.env` updated with `SKILL_AGENT_EX3_BASE_URL=http://localhost:11500/v1`
and `SKILL_AGENT_OPENAI_MODEL=google/gemma-4-31B-it`. Azure lines commented out.

### 2. Tool calling
First request to skillful-agent failed: `status 400: "auto" tool choice requires
--enable-auto-tool-choice and --tool-call-parser to be set`. Added both to
`extra_flags`. Parser: `pythonic` (generic fallback — Gemma has no named parser
in vLLM 0.21).

### 3. Context window investigation on a100q
After model load, only ~3.89 GiB remains for KV cache on 2× A100 40GB.
- 32k, 64k, 128k: all OOM (WorkerProc crash)
- 32k + `--cpu-offload-gb 20`: also OOM — base KV cache too small
Hard ceiling: ~8k on a100q for this model.

### 4. serve.py robustness fixes
Multiple bugs surfaced during the context experiments:
- **False-positive health check**: temp tunnel on 11501 hit stale jobs from prior
  runs. Fixed: poll the real tunnel on 11500 directly.
- **`ExitOnForwardFailure=yes`**: killed the tunnel before vLLM opened its port.
  Changed to `no` + auto-restart if tunnel dies.
- **Crash detector**: polls log every 3 health cycles (~24s). Bails immediately
  on crash patterns rather than waiting full 900s timeout.
- **Stale job cleanup**: `serve.py` now cancels all `llm-serve` jobs at startup.
- **Partition reason display**: PENDING dots now show Slurm reason code inline.

### 5. gh200q ARM venv
gh200q (GH200 96GB HBM3) was idle. Built `venv-gh200` on gh002 with
`pip install vllm` (aarch64 wheel). Hit two issues:
- `curand.h` missing: fixed by `CUDA_HOME=/usr/local/cuda-12.6` in sbatch.
- PartitionOption now carries `venv` and `tp` fields; a single model entry
  covers both gh200q and a100q with different configs.

### 6. Context on gh200q
After model load, ~5.34 GiB for KV cache on GPU. 128k needs 110 GiB → OOM.
32k + `--cpu-offload-gb 30` → **works**. CUDA graphs compiled, API responding.

## Current recommended setup
- **gh200q**: 32k context, tool calling, idle node, ARM venv.
- **a100q**: fallback, 8k context only, often queued.
- **hgx2q**: occupied; would give 128k+ when free.

## Key lessons
- 31B BF16 at 128k needs ~110 GiB KV cache — impossible on any single available GPU.
- CPU offload on GH200 is practical: hundreds of GiB of CPU RAM available, 30 GiB costs nothing.
- GH200 is aarch64 — separate venv required, first-run flashinfer JIT takes ~5 min but caches.
- Auto stale-job cleanup is essential: failed/interrupted runs leave jobs running.

## Final iteration of the day — 64k context confirmed

Pushed context further by increasing CPU offload. Findings:

- `--cpu-offload-gb 60` → 32k worked (we'd already proven this with 30 GB offload).
- `--cpu-offload-gb 120` + 128k → OOM, but vLLM helpfully reports `Available KV cache memory: 60.25 GiB` and `estimated maximum model length is 71760`.
- `--cpu-offload-gb 120` + 64k → ✅ works.

Settled on **64k as the production config**. 128k would need ~240 GB CPU offload — the GH200 node has the CPU RAM in principle but worth trying only if a workload actually demands it.

## Tunnel-detection bugs

Several false-positive issues with the readiness check hurt this session:

1. The original "vLLM ready" check used a temp tunnel on port 11501. Stale tunnels from prior jobs returned 200 instantly. Fixed by running curl on the eX3 login node directly against `node:28811`.
2. The login-node curl returned a string like `READY` or `NOT_READY`. The Python check `"READY" in result.stdout` was true for BOTH because `"NOT_READY".contains("READY")` is true. Renamed sentinels to `VLLM_IS_UP` / `VLLM_NOT_UP`.
3. Tunnel binding ≠ tunnel forwarding: SSH happily opens a local listener even when the remote port isn't reachable. Now we curl through the tunnel after opening it, retry up to 5 times, with `ControlMaster=no` so we don't share a tangled connection.
4. The status display printed dots regardless of what vLLM was actually doing. Replaced with a single updating line that shows the latest log message from the compute node.
