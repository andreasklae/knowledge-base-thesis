# 2026-05-23 — Gemma 4 31B-it on eX3, end-to-end

Got the self-hosted LLM endpoint working end-to-end. Gemma 4 31B-it on 2× A100 40GB (a100q, node n014), served by vLLM 0.21.0 on port 28811, tunneled to laptop `localhost:11500`. Coherent chat completions verified at ~23:14.

## What happened

Started by debugging the SSH tunnel that failed last session. Could not reproduce — `ssh -G ex3` showed ControlMaster off, `ssh -v -N -L 11500:n014:28811 ex3` bound both IPv4 and IPv6, `lsof` confirmed the listener. Treated the prior failure as transient.

Tried to expand from 0.5B test model to something useful. Briefly ran Qwen2.5-14B-Instruct end-to-end on n014 to confirm the pipeline still worked at a non-trivial size — it did.

Detoured into a multi-model fantasy (Qwen for code, Gemma for experiments, both behind LiteLLM). Dropped it as too much overhead for the actual need. Decided: one model per session, chosen at startup.

Explored bigger partitions for bigger models. `sinfo` showed g002 (hgx2q, 8× A100 80GB) mixed and gh001/gh002 (gh200q, 1× H100 96GB each) idle. Tried gh200q first since it was idle — landed on gh001, ran `uname -m`, got `aarch64`. The existing x86_64 venv is useless there. Backed out. Tried hgx2q — both `--gres=gpu:2` and `--gres=gpu:1` queued indefinitely. `squeue -p hgx2q` showed two users (`muhammady`, `uxue`) holding all 8 GPUs with multi-day time limits. Gave up on hgx2q for now and fell back to a100q with TP=2.

Settled on Gemma 4 31B-it. Originally aimed for Gemma 3 27B before being corrected: Gemma 4 exists (post my knowledge cutoff), has a 31B dense and a 26B A4B MoE, and is well-supported in vLLM 0.21.0 (gemma4.py, gemma4_mm.py, gemma4_mtp.py present).

Then a series of launches and errors, each fix one flag deeper:

1. `vllm serve` without env overrides → `PermissionError` writing to `/global/D1/huggingface/cache/hub`. Cluster-wide cache is read-only.
2. Set `HF_HOME` only → same `PermissionError`. `HF_HUB_CACHE` env var (set system-wide) overrode it.
3. Override `HF_HUB_CACHE` and `HF_ASSETS_CACHE` explicitly → cache permission fixed. Download proceeded. New error: `ValueError: Chunked MM input disabled but max_tokens_per_mm_item (2496) is larger than max_num_batched_tokens (2048)`. Gemma 4 is multimodal; vLLM force-disables chunked MM input for bidirectional-attention models, and the default batch token budget is too small.
4. Add `--max-num-batched-tokens 8192` → past that error. New error: KV cache memory shortage. 3.44 GiB needed for 8192 context, 2.71 GiB available. The 31B BF16 weights and activations eat most of the 40 GB.
5. Add `--gpu-memory-utilization 0.95` (default 0.9) → past KV cache error, engine starts, `Application startup complete`. Curl through tunnel returns coherent chat. Done.

## Side notes

- Leaked HF token in chat via `env | grep -iE 'hf_|huggingface|transformers'`. Need to revoke and regenerate. Logged this as a todo on the module page.
- SSH password prompt on every tunnel reconnect is going to be annoying. Set up `ssh-copy-id ex3` next.
- The Karpathy-style one-step-at-a-time debugging pattern paid off here. Every error message named the fix; the only question was which flag to add next.

## Decisions made

- Single model per session, not multi-model. Decide at startup.
- Drop GH200 for now until a separate aarch64 venv exists.
- Drop hgx2q until the long-running jobs clear.
- For the wrapper: sbatch (not srun), single Python script (not package). Earlier choice from AskUserQuestion; reaffirmed.
- Default model: Gemma 4 31B-it on a100q with TP=2.
